Foundry Master Spec 35D

Context Anchoring, Spec-State Alignment, and LLM Execution Guardrails

Preface

Foundry is an LLM-first framework. That means the framework should not merely tolerate AI-assisted development; it should actively structure it so that intent, state, and reasoning remain durable across sessions, tools, and models.

At present, most LLM-assisted development workflows suffer from the same core failure mode: important context lives in chat history instead of in repository artifacts. This causes drift, duplicated reasoning, forgotten tradeoffs, silent divergence from specs, and fragile handoffs between models or sessions.

This spec introduces a first-class, framework-supported context anchoring system for feature development.

The system separates three concerns:
	1.	Intent — what should be built
	2.	State — what is currently built
	3.	Reasoning history — why it was built that way

These concerns will be represented as three explicit feature documents:
	•	docs/features/<feature-name>.spec.md
	•	docs/features/<feature-name>.md
	•	docs/features/<feature-name>.decisions.md

This spec also introduces CLI tooling, verification rules, enforcement behavior, and scaffolded contributor guidance so that both the Foundry repository and generated Foundry applications can use this system consistently.

This is not a model-specific feature. It must work equally well with ChatGPT, Codex, Claude, Gemini, and future tools.

⸻

Goals

Implement a repository-native context anchoring system that:
	•	preserves feature intent across sessions
	•	preserves a permanent append-only record of decisions and reasoning
	•	keeps current implementation state visible and resumable
	•	detects and reports spec/state drift
	•	provides CLI tooling for inspection, validation, and repair guidance
	•	enables LLMs to refuse to proceed when required context artifacts are missing or invalid
	•	ships as part of Foundry’s out-of-the-box developer experience

⸻

Non-Goals

This spec does not:
	•	attempt to summarize or compact decision history
	•	require a database or non-file storage for feature context
	•	require integration with any single model vendor
	•	attempt to fully automate product management or roadmap creation
	•	replace existing Foundry compiler, verifier, or graph systems

Preface

	•	introduce advanced natural language inference
	•	require perfect semantic understanding
	•	rely on LLMs to determine alignment
	•	update scaffold docs yet
	•	integrate with inspect or verify flows yet

⸻

Mismatch Rules

A feature is in mismatch when one or more of the following conditions are true:
	1.	The spec describes requirements not reflected anywhere in:
	•	Current State
	•	Open Questions
	•	Next Steps
	2.	The state document describes current behavior or scope that is not grounded in:
	•	the spec
	•	or a decision entry that explains divergence
	3.	Acceptance criteria appear omitted from state tracking entirely.
	4.	Divergence appears to exist without a corresponding decision entry.

This does not require perfect semantic interpretation. Use deterministic heuristics that are explainable and testable.

⸻

Heuristic Guidance

Keep the initial alignment engine simple and explicit.

Acceptable early heuristics may include:
	•	comparing normalized bullet items across relevant sections
	•	checking whether acceptance criteria phrases appear in state sections
	•	checking whether divergence markers are referenced in decisions
	•	identifying obvious orphaned state claims

Do not claim strong semantic certainty beyond what the implementation can justify.

Issues should be framed as:
	•	mismatch
	•	possible mismatch
	•	unsupported state claim
	•	untracked spec requirement
	•	missing decision reference

Be honest and deterministic.

⸻

CLI Command

Command:

php bin/foundry context check-alignment --feature=<feature>
php bin/foundry context check-alignment --feature=<feature> --json

Optional support for --all is allowed if easy and consistent, but single-feature support is required.

⸻

JSON Output Contract

Required top-level shape:

{
  "status": "ok|warning|mismatch",
  "feature": "event-bus",
  "issues": [
    {
      "code": "untracked_spec_requirement",
      "message": "Acceptance criteria item is not reflected in Current State, Open Questions, or Next Steps.",
      "spec_section": "Acceptance Criteria",
      "state_section": null,
      "decision_reference_found": false
    }
  ],
  "required_actions": []
}

Keep keys stable and deterministic.
