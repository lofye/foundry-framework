# 002.001-marketplace-auth-runtime-contracts

## Purpose

Tighten the Marketplace identity and authentication runtime introduced by
`002-marketplace-identity-and-authentication.md`.

The original 002 spec established the first-pass Marketplace identity/auth CLI and runtime.
This follow-up spec formalizes the runtime contracts so later Marketplace specs can safely depend on them.

This spec must preserve the public behavior already implemented by 002 unless the existing behavior is clearly incomplete, unsafe, nondeterministic, or inconsistent with this spec.

---

## Background

Marketplace 002 introduced:

- `foundry login`
- `foundry logout`
- `foundry whoami`
- local identity/token storage
- a Marketplace auth runtime
- deterministic CLI output expectations

However, it did not fully specify:

- token lifecycle
- credential store semantics
- malformed state handling
- security expectations
- deterministic JSON shapes
- failure codes
- verification/inspection requirements
- how future entitlement/purchase/MCP flows should consume auth state

This spec adds those missing contracts.

---

## Scope

Implement or tighten the framework-side Marketplace auth runtime.

Framework repo responsibilities:

- contracts
- deterministic CLI behavior
- local credential storage abstraction
- local token/identity state validation
- auth state inspection
- auth state verification
- safe API-client credential access

Website repo responsibilities remain out of scope:

- hosted login UI
- real account creation
- production token issuing
- production session management
- payment/account dashboards

The framework module may provide file-backed/local reference behavior, but it must not become the canonical hosted account application.

---

## Goals

1. Formalize Marketplace credential storage behavior.
2. Formalize token lifecycle behavior.
3. Standardize JSON output contracts for auth commands.
4. Ensure malformed auth state fails closed.
5. Ensure secrets never leak through inspect/verify/whoami output.
6. Provide deterministic auth verification.
7. Make auth state safe for later entitlement, purchase, MCP, and Generate integrations.

---

## Non-Goals

- Do not implement website signup/login UI.
- Do not implement production OAuth.
- Do not implement payment provider logic.
- Do not implement entitlements.
- Do not implement license activation.
- Do not require OS keychain integration.
- Do not introduce nondeterministic token generation into tests.
- Do not store secrets in logs, inspect payloads, or verifier output.

---

## Runtime Boundary

The Marketplace framework module defines the client/protocol side of Marketplace auth.

It may include:

- `MarketplaceIdentityStore`
- `MarketplaceAuthService`
- `MarketplaceApiClient` or equivalent
- deterministic CLI commands
- local credential persistence
- token inspection without secret disclosure
- verification of local auth state

It must not include:

- production hosted account database
- public website session controller
- payment-provider session logic
- account dashboard rendering

## Required Runtime Concepts

### Marketplace Identity

Canonical identity object:

```json
{
  "id": "user_123",
  "email": "user@example.com",
  "name": "Optional Name",
  "created_at": "2026-01-01T00:00:00Z"
}
```

Rules:

- `id` is required.
- `email` is required.
- `name` is optional and may be `null`.
- `created_at` must be an ISO-8601 string when present.
- JSON serialization must use stable key ordering.
- Identity output must never include access tokens, refresh tokens, API keys, or license keys.

### Marketplace Credentials

Canonical local credential record:

```json
{
  "token_type": "bearer",
  "access_token": "secret_value",
  "expires_at": null,
  "user": {
    "id": "user_123",
    "email": "user@example.com",
    "name": null,
    "created_at": "2026-01-01T00:00:00Z"
  }
}
```

Rules:

- `access_token` is required for authenticated state.
- `token_type` defaults to `bearer`.
- `expires_at` may be `null`.
- `user.id` and `user.email` are required.
- malformed credential files must not authenticate.
- partial credentials must not authenticate.
- missing credentials produce unauthenticated state, not a crash.

## Credential Store Contract

Introduce or harden a replaceable credential store abstraction.

Suggested name:

```text
MarketplaceCredentialStore
```

If an existing class already serves this purpose, update it rather than duplicating it.

Required operations:

```text
read(): MarketplaceCredentialState
write(credentials): void
clear(): void
exists(): bool
inspect(): array
verify(): VerificationResult
```

The exact PHP method names may differ if the repository conventions require it, but the responsibilities must exist and be tested.

### Store Requirements

The store must:

- be root-aware when used inside temp projects/tests.
- avoid process-CWD assumptions.
- support atomic replacement where practical.
- fail closed on malformed data.
- never leak raw secrets in inspection output.
- be deterministic in JSON serialization.
- be replaceable by a future OS-keychain-backed implementation.
- avoid global mutable singleton state.

### File-Backed Reference Store

A file-backed implementation is acceptable for now.

Recommended path:

```text
.foundry/marketplace/auth.json
```

If the repo already uses a different path, keep the existing path only if documented in Marketplace context.

File-backed behavior:

- parent directory must be created when needed.
- credentials must be written deterministically.
- logout must remove the credential file or replace it with a known unauthenticated state.
- unreadable/malformed files must produce structured fail-closed results.
- write failures must produce deterministic errors.

## Token Lifecycle

### Login

`foundry login` must:

- accept the already-supported login mode from Marketplace 002.
- persist credentials through the credential store.
- replace existing credentials atomically.
- verify identity after login when possible.
- return deterministic output.

### Logout

`foundry logout` must:

- clear local Marketplace credentials.
- be idempotent.
- return success even when already logged out unless a storage error occurs.
- not leave partial credential state.

### Whoami

`foundry whoami` must:

- read local auth state.
- return authenticated identity when valid.
- return unauthenticated state when missing.
- return fail-closed structured errors when malformed/invalid state is encountered.

### Expiration

If `expires_at` is present:

- expired credentials must not authenticate.
- expired credentials should be visible as expired without revealing token values.
- expiration comparison must be deterministic in tests by allowing injected clock or controlled time source if needed.

If no clock abstraction exists, add the smallest one necessary.

---

## CLI Commands

The following commands must be covered:

```bash
foundry login
foundry logout
foundry whoami
```

JSON mode must be supported:

```bash
foundry login --json
foundry logout --json
foundry whoami --json
```

Plain text output must also remain deterministic.

## JSON Output Contracts

### Authenticated Whoami

```json
{
  "authenticated": true,
  "user": {
    "id": "user_123",
    "email": "user@example.com",
    "name": null,
    "created_at": "2026-01-01T00:00:00Z"
  }
}
```

### Unauthenticated Whoami

```json
{
  "authenticated": false,
  "user": null
}
```

### Expired Credentials

```json
{
  "authenticated": false,
  "reason": "expired",
  "user": {
    "id": "user_123",
    "email": "user@example.com"
  }
}
```

The user field may be omitted if the implementation does not retain safe identity details after expiry, but behavior must be deterministic and tested.

### Malformed Credentials

```json
{
  "status": "error",
  "code": "MARKETPLACE_AUTH_STATE_INVALID",
  "authenticated": false
}
```

### Logout

```json
{
  "status": "ok",
  "authenticated": false
}
```

### Login Success

```json
{
  "status": "ok",
  "authenticated": true,
  "user": {
    "id": "user_123",
    "email": "user@example.com"
  }
}
```

## Error Codes

Use deterministic error codes.

Required codes, or equivalent existing names if already established:

```text
MARKETPLACE_AUTH_FAILED
MARKETPLACE_AUTH_STATE_INVALID
MARKETPLACE_AUTH_STORAGE_UNREADABLE
MARKETPLACE_AUTH_STORAGE_UNWRITABLE
MARKETPLACE_AUTH_EXPIRED
MARKETPLACE_AUTH_NOT_AUTHENTICATED
```

Do not introduce vague errors like `ERROR`, `FAILED`, or `UNKNOWN` when a specific cause is known.

---

## Secret Redaction

The following must never appear in normal CLI output, inspect output, verify output, exceptions, test snapshots, or logs:

- access token values
- refresh token values
- API keys
- license keys
- bearer token headers

Safe output may include:

- token present: true/false
- token type
- expires_at
- user id
- user email
- credential path, if not sensitive

---

## Marketplace API Client Integration

If a Marketplace API client exists, it must consume credentials through the credential store/auth service and not read credential files directly.

Rules:

- authenticated requests use the current valid token.
- unauthenticated requests fail closed when auth is required.
- expired credentials must not be used.
- malformed credentials must not be used.
- error responses must be deterministic.

---

## Verify / Inspect Integration

If `verify marketplace --json` exists, it must include auth-state validation or explicitly report auth as not configured.

If `inspect marketplace --json` exists, it must include safe auth inspection without leaking secrets.

Example inspect shape:

```json
{
  "auth": {
    "configured": true,
    "authenticated": true,
    "user": {
      "id": "user_123",
      "email": "user@example.com"
    },
    "token": {
      "present": true,
      "expires_at": null
    }
  }
}
```

If auth state is malformed:

```json
{
  "auth": {
    "configured": true,
    "authenticated": false,
    "status": "invalid",
    "code": "MARKETPLACE_AUTH_STATE_INVALID"
  }
}
```

---

## Determinism Requirements

- Stable key ordering in JSON.
- Stable list ordering.
- No timestamps in outputs unless part of persisted/auth state.
- No random IDs in tests unless injected/fixed.
- No CWD-dependent credential discovery.
- No environment leakage in deterministic output.

---

## Testing Requirements

Add focused tests for:

### Credential Store

- missing credential file.
- valid credential file.
- malformed JSON.
- missing access token.
- missing user id.
- missing user email.
- expired credential.
- unreadable credential file if practical in the environment.
- write failure if practical without platform-specific flakiness.
- logout idempotency.
- credential replacement.

### Auth Service

- login persists valid credentials.
- login replaces previous credentials.
- logout clears credentials.
- whoami authenticated.
- whoami unauthenticated.
- whoami expired.
- malformed state fails closed.

### CLI

- `foundry whoami --json` authenticated.
- `foundry whoami --json` unauthenticated.
- `foundry logout --json` idempotent.
- deterministic plain text for whoami/logout.
- no secret leakage.

### Verify/Inspect

- inspect marketplace redacts tokens.
- verify marketplace reports malformed auth state deterministically.
- verify marketplace passes when auth is absent but not required.

---

## Acceptance Criteria

- Marketplace auth runtime has documented deterministic contracts.
- Credential storage is root-aware and replaceable.
- malformed credential state fails closed.
- expired credentials do not authenticate.
- logout is idempotent.
- login replaces credentials atomically where practical.
- auth inspect/verify output never leaks secrets.
- JSON outputs are deterministic.
- all tests and coverage gates pass.
