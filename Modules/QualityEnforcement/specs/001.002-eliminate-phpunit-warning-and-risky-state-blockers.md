# Execution Spec: 001.002-eliminate-phpunit-warning-and-risky-state-blockers

## Feature
- quality-enforcement

## Purpose
- Restore a truly clean strict PHPUnit baseline for the repository.
- Eliminate the current warning, risky-state, and stale-assertion blockers that keep `php vendor/bin/phpunit` exiting nonzero under the repository’s intentionally strict PHPUnit configuration.
- Ensure the hard completion gate added in `quality-enforcement/001` and `001.001` can succeed honestly without weakening PHPUnit strictness.

## Scope
- Fix the currently identified nonzero-exit blockers in the repository test suite and the framework/test code that causes them.
- Preserve the existing strict PHPUnit configuration.
- Keep this work narrowly focused on making the full repository suite exit cleanly under current intended rules.
- Do not relax PHPUnit standards to make the suite green.

## Constraints
- Do not weaken `phpunit.xml` by disabling:
  - `failOnWarning`
  - `failOnRisky`
  - strict output behavior
- Prefer the smallest correct fix at each blocker site.
- Keep fixes deterministic and explicit.
- Do not introduce broad refactors unless a blocker cannot be solved safely otherwise.
- Do not change the meaning of existing quality-enforcement rules.
- The final result must make strict full-suite execution a trustworthy signal.

## Inputs

Expect inputs such as:
- current `phpunit.xml`
- the repository’s strict PHPUnit behavior
- the diagnosed blocker list from the latest full-suite investigation
- current full-suite and coverage commands
- affected framework classes and tests

If any critical input is missing:
- fail clearly and deterministically
- do not relax the config as a substitute
- do not pretend the suite is clean

## Requested Changes

### 1. Preserve Strict PHPUnit Configuration

Keep the repository’s current strict PHPUnit semantics intact.

This spec must not “fix” the problem by changing PHPUnit from strict to permissive.

In particular, do not disable or weaken:
- warning failures
- risky-test failures
- strict output handling

### 2. Fix the Known Ordinary Test Failure

Fix the stale assertion in:

- `tests/Integration/CLIInitAppCommandTest.php`

The change must align the test with current intended scaffolded output and repository guidance.

This is a test-code issue, not a config issue.

### 3. Fix the Known Risky Tests

Fix the three risky tests in:

- `tests/Unit/FirstRunServiceTest.php`

These tests currently become risky because tested code prints output under strict output rules.

Use the smallest correct approach, such as:
- capturing/asserting expected output
- otherwise structuring the tests so the output is not “unexpected” under strict mode

Do not suppress the output rules globally.

### 4. Fix Warning-Producing Framework Behavior

Apply the smallest correct framework fixes for the currently diagnosed warning sources:

#### A. `CompileCacheInspector`
Fix the duplicate child-directory removal / cleanup path so it no longer emits repeated `rmdir(...): No such file or directory` warnings.

#### B. `ImplementationQualityGateService`
Guard cleanup of temporary coverage artifacts so it no longer emits repeated `unlink(...): No such file or directory` warnings.

#### C. `ContextPlanningService`
Make stub loading fail cleanly and intentionally when the expected stub file is missing or unreadable, rather than emitting PHP warnings before controlled failure handling.

These are framework behavior issues and should be fixed in framework code, not by muting warnings in tests.

### 5. Fix Warning-Producing Test Helpers

Apply the smallest correct cleanup to the currently identified test-helper warning sites, including cases like:
- repeated `mkdir(): File exists`
- `unlink(...): No such file or directory`
- `rmdir(...): Directory not empty`

These should be solved by making the helper/test setup-teardown behavior correct and idempotent under repeated strict runs.

Do not hide them through global error suppression.

### 6. Require a Clean Full-Suite Exit

The acceptance target for this spec is not merely “tests mostly pass.”

The required outcome is:

```bash
php vendor/bin/phpunit
```

must exit `0` under the repository’s current strict configuration.

### 7. Keep the Coverage Gate Clean Too

After the strict full-suite exit is fixed, the coverage command must still succeed under the current quality-enforcement rules:

```bash
php -d xdebug.mode=coverage vendor/bin/phpunit --coverage-text
```

This spec must not fix one gate while breaking the other.

### 8. Keep the Fix Surface Honest and Bounded

This spec is a blocker-removal pass, not a broad modernization sweep.

Only fix:
- the known failing assertion
- the known risky output behavior
- the known warning-producing framework behavior
- the known warning-producing test-helper behavior
- any directly adjacent minimal issues discovered while making those fixes

Do not broaden into unrelated cleanup.

### 9. Tests

Add or update focused coverage proving:

- the previously failing assertion is corrected
- the risky FirstRun tests now pass cleanly under strict output behavior
- `CompileCacheInspector` cleanup no longer emits warning-producing double-removal behavior
- `ImplementationQualityGateService` cleanup no longer emits warning-producing missing-file unlinks
- `ContextPlanningService` stub loading fails cleanly without raw PHP warning leakage
- warning-producing test helpers are cleaned up correctly
- `php vendor/bin/phpunit` exits `0`
- `php -d xdebug.mode=coverage vendor/bin/phpunit --coverage-text` still exits successfully

## Non-Goals
- Do not weaken `phpunit.xml`.
- Do not redesign the test framework.
- Do not refactor unrelated code just because it is nearby.
- Do not lower the quality-enforcement threshold.
- Do not broaden this into a general repository cleanup campaign.
