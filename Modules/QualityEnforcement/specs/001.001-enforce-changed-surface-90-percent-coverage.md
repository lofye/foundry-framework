# Execution Spec: 001.001-enforce-changed-surface-90-percent-coverage

## Feature
- quality-enforcement

## Purpose
- Extend quality-enforcement so changed/touched implementation surfaces must also meet the required coverage threshold before completion is allowed.
- Close the temporary gap left by `001-require-full-suite-and-90-percent-coverage-before-completion`, which enforced global line coverage but only reported changed-surface coverage as unsupported.
- Prevent implementations from passing the quality gate while the files they actually changed remain under-covered.

## Scope
- Add deterministic changed-surface coverage computation to the existing quality gate.
- Enforce changed-surface line coverage >= 90% as a hard completion requirement.
- Reuse the existing full-suite and coverage execution path introduced in `quality-enforcement/001`.
- Keep this focused on changed-surface detection, coverage attribution, and enforcement.

## Constraints
- The enforcement must remain deterministic and machine-readable.
- It must continue to rely on the full PHPUnit suite and coverage run as the source of truth.
- It must not silently fall back to “unsupported” once this spec is implemented.
- Prefer explicit changed-file computation over fuzzy heuristics.
- Do not weaken the already-enforced global line coverage >= 90% requirement.
- Do not treat untouched files as part of the changed-surface gate.
- Do not require developers to manually specify changed files during normal Foundry-owned implementation workflows if the repository can derive them deterministically.

## Inputs

Expect inputs such as:
- full-suite coverage output
- implementation-time changed/touched file list
- repository working tree or implementation action list
- current `ImplementationQualityGateService`
- Foundry-owned implementation/stabilization workflows

If any critical input is missing:
- fail clearly and deterministically
- do not pretend changed-surface coverage passed
- do not silently downgrade enforcement to advisory mode

## Requested Changes

### 1. Compute Changed/Touched Surface Deterministically

Implement a deterministic way to identify the source files changed by the current implementation run.

Preferred sources, in order of trustworthiness:
1. files changed by the Foundry-owned implementation workflow itself
2. repository diff against the pre-run state
3. another existing deterministic repository-owned mechanism

The implementation must choose the smallest trustworthy approach already available in the repository.

Changed-surface computation must:
- include modified existing source files
- include newly added source files when they are part of the implementation
- exclude unrelated unmodified files
- exclude docs-only changes unless they affect covered source under enforcement
- be stable across repeated runs against the same working tree

### 2. Attribute Coverage to Changed Files

Use the coverage result from the required full-suite coverage run to compute line coverage for each changed/touched source file under enforcement.

At minimum, this must work for repository-owned PHP source files that are expected to be covered by PHPUnit/Xdebug coverage data.

The quality gate must identify:
- which changed files are below 90% line coverage
- their measured line coverage
- whether the changed-surface gate passes overall

### 3. Enforce Changed-Surface Threshold

Add a hard failure when one or more changed/touched files under enforcement are below:

- line coverage >= 90%

Once implemented, completion must fail if:
- global line coverage is below 90%, or
- any changed/touched enforced source file is below 90%

### 4. Keep Reporting Machine-Readable

Extend the existing quality-gate output so it clearly reports:

- changed files examined
- per-file changed-surface coverage
- under-covered changed files
- changed-surface threshold required
- changed-surface pass/fail status

The JSON/reporting must be deterministic and suitable for CLI and workflow consumers.

### 5. Fail Clearly When Attribution Is Impossible

If the gate cannot determine changed files or cannot attribute coverage to them in a trustworthy way:
- fail clearly
- report why changed-surface enforcement could not be computed
- do not report success

This spec removes the temporary “unsupported” escape hatch.

### 6. Integrate With Existing Quality Gate

Reuse and extend the implementation quality gate introduced in `quality-enforcement/001`.

Do not create a second, disconnected enforcement path.

The same finalization/workflow gate should now enforce:
- full-suite test execution
- global line coverage >= 90%
- changed-surface line coverage >= 90%

### 7. Tests

Add focused coverage proving:

- changed files are computed deterministically
- changed files with >= 90% coverage pass
- changed files with < 90% coverage fail
- newly added source files are included correctly
- unrelated untouched files do not affect changed-surface enforcement
- attribution/reporting is deterministic and machine-readable
- the old “unsupported” state is no longer accepted after this spec
- all relevant existing quality gate tests still pass

## Non-Goals
- Do not redesign PHPUnit coverage collection.
- Do not change the global >= 90% line coverage rule except to keep enforcing it.
- Do not introduce method/class coverage as the required changed-surface metric in this spec.
- Do not broaden this into a general diff-quality score.
- Do not silently auto-generate tests in this spec.

## Authority Rule
- Completion must fail unless changed/touched implementation surfaces under enforcement meet the required 90% line coverage threshold.
- Changed-surface enforcement must be deterministic, explicit, and machine-readable.
- Unsupported changed-surface reporting is no longer sufficient after this spec.

## Completion Signals
- Changed/touched implementation surfaces are computed deterministically.
- Per-file changed-surface coverage is reported.
- Under-covered changed files fail the quality gate.
- The temporary “unsupported changed-surface coverage” status is eliminated.
- Full-suite + global + changed-surface enforcement now all work together.
- All tests pass.

## Post-Execution Expectations
- Foundry’s quality gate now enforces the rule you actually want: changed code must be well covered, not just the repository overall.
- It becomes much harder for low-coverage new work to hide inside a repo with acceptable aggregate coverage.
- Implementation completion becomes stricter and more trustworthy.
