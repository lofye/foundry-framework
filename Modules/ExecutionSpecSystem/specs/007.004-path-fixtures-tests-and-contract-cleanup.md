# Execution Spec: 007.004-path-fixtures-tests-and-contract-cleanup

## Feature

- execution-spec-system

## Purpose

Clean up fixtures, tests, snapshots, contract expectations, deterministic CLI outputs, and validation examples so they match the final canonical feature documentation layout.

This spec closes the test and contract gap after the documentation path migration and repairs any stale expectations left behind by implementing `007.001-correct-feature-docs-layout.md` early.

## Depends On

- `007-modular-docs-feature-layout.md`
- `007.001-correct-feature-docs-layout.md`
- `007.002-agent-facing-doc-path-contracts.md`
- `007.003-readmes-stubs-and-public-docs-path-contracts.md`

## Canonical Path Contract

Feature context files:

```text
docs/features/<feature>/<feature>.spec.md
docs/features/<feature>/<feature>.md
docs/features/<feature>/<feature>.decisions.md
```

Feature execution specs and drafts:

```text
docs/features/<feature>/specs/<id>-<slug>.md
docs/features/<feature>/specs/drafts/<id>-<slug>.md
```

Feature implementation plans:

```text
docs/features/<feature>/plans/<id>-<slug>.md
```

Feature support files:

```text
docs/features/README.md
docs/features/implementation-log.md
```

## Scope

Update all test and contract surfaces that encode feature-doc paths, including at minimum:

- tests that create, discover, validate, or assert docs paths
- test helpers that write feature context files, execution specs, drafts, plans, or implementation logs
- fixtures containing old path patterns
- snapshots or expected output files
- JSON contract fixtures
- CLI output expectations
- validation error examples
- completion output expectations
- scaffold output expectations
- smoke/demo app fixtures
- docs import/export tests, if present

## Required Cleanup

1. Ensure tests create feature context files with `<feature>` stems, not `<id>-<slug>` stems.
2. Ensure execution spec tests create specs under `docs/features/<feature>/specs/`.
3. Ensure draft spec tests create drafts under `docs/features/<feature>/specs/drafts/`.
4. Ensure plan tests, if any exist before `008`, either:
   - use `docs/features/<feature>/plans/`, or
   - are deferred until `008` introduces first-class plan validation.
5. Ensure implementation-log tests use `docs/features/implementation-log.md`.
6. Ensure validators reject old active paths deterministically where applicable.
7. Ensure invalid-path fixtures are clearly named or commented as invalid-path coverage.
8. Remove obsolete fixture directories that are no longer used.
9. Keep deterministic ordering in all updated outputs.

## Invalid Active Test Assumptions

Tests and fixtures must not treat these as valid active paths:

```text
docs/<feature>/
docs/specs/
docs/features/<id>-<slug>.md
docs/features/<id>-<slug>.spec.md
docs/features/<id>-<slug>.decisions.md
docs/features/<feature>/<id>-<slug>.md
docs/features/<feature>/<id>-<slug>.spec.md
docs/features/<feature>/<id>-<slug>.decisions.md
```

They may appear only in tests that explicitly assert invalid path handling.

## Implementation Steps

1. Search tests, fixtures, snapshots, and contract files for stale paths.
2. Update fixture directory structures to mirror the canonical layout.
3. Update helper methods that write feature docs, specs, drafts, plans, or logs.
4. Update CLI expected outputs to emit canonical paths.
5. Update JSON expected outputs to emit canonical paths.
6. Update validation error tests to expect canonical path guidance.
7. Add regression tests where gaps are found, especially for:
   - feature context filename stems
   - implementation-log path
   - rejection or non-recognition of old paths
8. Remove unused obsolete fixtures.
9. Do not weaken assertions to make the suite pass.

## Acceptance Criteria

- Tests no longer rely on old paths as valid active locations.
- Fixture helpers create feature context files at canonical paths.
- Fixture helpers create execution specs and drafts at canonical paths.
- Implementation-log fixtures and assertions use `docs/features/implementation-log.md`.
- Snapshot and JSON fixtures use canonical paths.
- CLI contract outputs remain deterministic.
- Validation errors describe the canonical path contract.
- Old path references remain only in explicitly invalid-path tests or historical migration fixtures.
- Full PHPUnit and coverage runs pass without warnings.

## Verification

Run:

```bash
php bin/foundry spec:validate --json
php bin/foundry verify context --json
php bin/foundry verify contracts --json
php bin/foundry verify graph --json
php bin/foundry verify pipeline --json
php vendor/bin/phpunit
php -d xdebug.mode=coverage vendor/bin/phpunit --coverage-text
```

Also run and inspect:

```bash
grep -R "docs/<feature>" tests fixtures snapshots 2>/dev/null || true
grep -R "docs/specs" tests fixtures snapshots 2>/dev/null || true
grep -R "docs/features/<id>-<slug>" tests fixtures snapshots 2>/dev/null || true
grep -R "docs/features/<feature>/<id>-<slug>" tests fixtures snapshots 2>/dev/null || true
```

Any remaining matches must be invalid-path fixtures, historical migration fixtures, or corrected.

## Implementation Log

Append the completion entry to `docs/features/implementation-log.md` only after all verification commands pass.
