# Execution Spec: 007.003-readmes-stubs-and-public-docs-path-contracts

## Feature

- execution-spec-system

## Purpose

Align human-facing documentation, README files, examples, public docs, and generated stubs with the final canonical feature documentation layout.

This spec follows `007.002-agent-facing-doc-path-contracts.md` and handles documentation surfaces that are primarily read by humans or copied into generated apps.

## Background

`007.001-correct-feature-docs-layout` may already have been implemented before this spec. If so, do not undo that correction. Treat `docs/features/<feature>/...` as already canonical and repair only stale wording, examples, generated docs, stubs, fixtures, and public documentation references.

## Depends On

- `007-modular-docs-feature-layout.md`
- `007.001-correct-feature-docs-layout.md`
- `007.002-agent-facing-doc-path-contracts.md`

## Canonical Path Contract

Feature context files:

```text
docs/features/<feature>/<feature>.spec.md
docs/features/<feature>/<feature>.md
docs/features/<feature>/<feature>.decisions.md
```

Feature execution specs and drafts:

```text
docs/features/<feature>/specs/<id>-<slug>.md
docs/features/<feature>/specs/drafts/<id>-<slug>.md
```

Feature implementation plans:

```text
docs/features/<feature>/plans/<id>-<slug>.md
```

Feature support files:

```text
docs/features/README.md
docs/features/implementation-log.md
```

## Scope

Update human-facing and generated documentation surfaces, including at minimum:

- `README.md`
- `APP-README.md`
- `docs/features/README.md`
- other docs under `docs/**/*.md` that describe feature docs, execution specs, drafts, plans, or implementation logs
- stubs under `stubs/**`
- scaffold docs copied into generated applications
- example apps or smoke/demo docs, if present
- CLI examples stored in docs or stubs
- diagrams, tables, and code blocks that show feature-doc paths

## Required Corrections

1. Replace active references to `docs/<feature>/...` with `docs/features/<feature>/...`.
2. Replace active references to `docs/specs/implementation-log.md` with `docs/features/implementation-log.md`.
3. Replace active references to `docs/specs/README.md` with `docs/features/README.md`.
4. Replace feature context examples using `<id>-<slug>` stems with `<feature>` stems.
5. Preserve execution spec examples using `<id>-<slug>` stems under `docs/features/<feature>/specs/`.
6. Preserve draft spec examples under `docs/features/<feature>/specs/drafts/`.
7. Add plan examples only under `docs/features/<feature>/plans/`.
8. Ensure docs explain the distinction between:
   - canonical feature context
   - feature-local execution specs
   - feature-local draft execution specs
   - feature-local implementation plans
   - global feature implementation log
9. Keep historical migration references only when explicitly labeled as historical.

## Invalid Active Documentation Patterns

The following must not appear as current instructions, examples, or scaffold output:

```text
docs/<feature>/
docs/specs/
docs/features/<id>-<slug>.md
docs/features/<id>-<slug>.spec.md
docs/features/<id>-<slug>.decisions.md
docs/features/<feature>/<id>-<slug>.md
docs/features/<feature>/<id>-<slug>.spec.md
docs/features/<feature>/<id>-<slug>.decisions.md
```

## Implementation Steps

1. Search `README.md`, `APP-README.md`, `docs/`, `stubs/`, examples, and scaffold sources for stale path references.
2. Update README guidance to describe the feature-local layout precisely.
3. Update APP README guidance so generated apps receive correct framework/app docs expectations.
4. Update stubs so newly generated files do not contain stale paths.
5. Update examples that show creating, validating, promoting, planning, or implementing execution specs.
6. Update diagrams, tables, and code blocks to use canonical paths.
7. Do not move non-feature docs directories.
8. Do not introduce fallback behavior for old paths.
9. Keep wording deterministic and direct.

## Acceptance Criteria

- `README.md` uses canonical paths for all active feature-doc guidance.
- `APP-README.md` uses canonical paths for all active feature-doc guidance where relevant.
- `docs/features/README.md` describes the final canonical layout, including plans.
- Stubs generate only canonical path guidance.
- Human-facing docs do not claim feature context files are named `<id>-<slug>.*.md`.
- Human-facing docs do not claim `docs/specs/` is canonical.
- Human-facing docs do not claim feature folders live directly under `docs/`.
- Any remaining stale paths are historical migration notes or invalid-path examples clearly marked as such.
- The documentation distinguishes `docs/features/implementation-log.md` from feature-local specs and plans.

## Verification

Run:

```bash
php bin/foundry spec:validate --json

php bin/foundry verify contracts --json
php bin/foundry verify graph --json
php bin/foundry verify pipeline --json
php vendor/bin/phpunit

```

Also run and inspect:

```bash
grep -R "docs/<feature>" README.md APP-README.md docs stubs examples 2>/dev/null || true
grep -R "docs/specs" README.md APP-README.md docs stubs examples 2>/dev/null || true
grep -R "docs/features/<id>-<slug>" README.md APP-README.md docs stubs examples 2>/dev/null || true
grep -R "docs/features/<feature>/<id>-<slug>" README.md APP-README.md docs stubs examples 2>/dev/null || true
```

Any remaining matches must be corrected or explicitly labeled as historical or invalid-path examples.
