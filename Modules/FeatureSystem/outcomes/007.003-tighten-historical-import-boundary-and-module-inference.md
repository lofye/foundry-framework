## Runtime Contracts

- `historical-specs:evidence --json` now emits deterministic top-level `canonical_transition` metadata (`legacy_label`, `canonical_path`, optional current canonical path, confidence) and deterministic per-era `counts`.
- Candidate rows now emit explicit import boundary metadata: `era`, `import_action`, and `canonical_transition_relative`.
- Candidate rows include `module_inference` evidence and deterministic alternatives ordering so pre-canonical module assignment remains auditable and reviewable.
- Canonical-era candidates (`Spec35D1+` by default transition) use link-or-review behavior: `link_existing` when an existing canonical spec path is discovered, otherwise deterministic `review`.

## Deterministic Outputs

- Candidate ordering remains stable by known legacy-order key, then source filename, then source segment index.
- Era/action classification is deterministic given source text plus optional anchor config.
- Transition/count payload keys are deterministic and timestamp-free.
- Paths in evidence-map output remain repository-relative (no absolute host paths).

## Decisions And Tradeoffs

- Kept the built-in `Spec35D1` transition anchor to provide safe defaults while allowing deterministic override via anchor configuration.
- Preserved non-importing evidence behavior (`historical-specs:evidence` remains prep-only) to avoid premature module/spec migration side effects.
- Used explicit review fallbacks for ambiguous, conflicting, or unresolved canonical-existing mappings instead of speculative automatic assignment.

## Reconstruction Notes

- Reconstruct by loading source historical files in deterministic order, segmenting candidates, then classifying each segment relative to the configured/default transition anchor.
- Apply module inference from text/path/git signals, preserve alternatives, and convert conflicts/low-confidence cases into deterministic `review` actions.
- Keep supporting-evidence sources explicit and non-importing so follow-up import specs can consume the map safely.
