- Implemented apply-mode writes for `_import/historical-specs/candidate-XXX/{spec.md,source.md,metadata.json}` with stale candidate directory cleanup for deterministic reruns.
- Added unit and integration coverage for multi-spec extraction, no-spec files, weird labels (`Spec 35D-2`), duplicate slug handling, low-confidence fallback extraction, dry-run no-write behavior, apply determinism, and missing-source errors.

## Runtime Contracts

- `historical-specs:extract` is a prep-only workflow command; it does not import candidates into `Modules/*`, write implementation-log entries for historical specs, or generate reconstruction notes.
- Candidate artifacts are written only under the target archive path as `candidate-XXX/spec.md`, `candidate-XXX/source.md`, and `candidate-XXX/metadata.json`.
- JSON/path output is repository-relative and deterministic, with no timestamp or absolute-path fields.

## Deterministic Outputs

- Source scan order is deterministic (sorted `.md`/`.txt` paths).
- Candidate numbering is deterministic (`candidate-%03d`) and stable for identical input.
- Metadata keys are written in fixed order: `original_file`, `candidate_id`, `detected_spec_label`, `detected_title`, `suggested_slug`, `suggested_module`, `confidence`, `notes`.
- Apply mode rewrites candidate directories deterministically by clearing stale `candidate-XXX` directories before writing current output.

## Decisions And Tradeoffs

- Extraction uses conservative heuristics and confidence labeling instead of attempting full semantic reconstruction during prep.
- `suggested_module` is intentionally emitted as `unknown` by default to avoid accidental ownership claims during archival extraction.
- Candidate reruns delete only prior `candidate-XXX` directories in the target path to preserve deterministic outputs without touching unrelated files.

## Reconstruction Notes

- Reconstruct by creating a sorted archive scan over `_import/raw-historical-specs` and splitting candidate sections using explicit `Spec`/`Execution Spec` heading heuristics plus title/purpose fallback when no hard boundaries exist.
- Keep the extraction contract prep-only: preserve extracted source text, emit cleaned candidate spec text, and emit best-effort metadata without touching module/runtime ownership.
- Wire the command into both CLI dispatch and API surface classification so `help` and command metadata remain deterministic.
