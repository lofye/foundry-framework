- Extended historical extraction metadata with segment-index and optional RESULT/FOLLOWUPS detection so multi-spec and optional output blocks remain explicit prep artifacts.

## Runtime Contracts

- `historical-specs:evidence` provides deterministic evidence-map generation over `_import/historical-specs` and does not import specs into `Modules/*`.
- Evidence-map candidate rows expose required fields: `candidate_id`, `source_file`, `legacy_label`, `legacy_order_key`, `detected_title`, `suggested_module`, `suggested_spec_path`, `implemented`, `confidence`, `evidence`, and `notes`.
- Legacy-order parsing supports mixed labels (for example `Spec 19FB`, `Spec 30C-2`, `Spec 35D7JA`) with stable sort keys and unknown-label fallback ordering.
- RESULT/OUTPUT and FOLLOWUPS sections are preserved as explicit extraction/evidence signals rather than hidden assumptions.

## Deterministic Outputs

- Candidate ordering is stable by known legacy order key, then source filename, then source segment index.
- Evidence-map JSON and markdown report output are stable and timestamp-free.
- Path fields are repository-relative; absolute local paths are not emitted.
- Unknown/ambiguous mappings remain explicit through confidence/evidence values instead of silent promotion.

## Decisions And Tradeoffs

- Chose a dedicated evidence command (`historical-specs:evidence`) instead of overloading extraction with full evidence-map logic, keeping extraction and ordering/evidence responsibilities separated.
- Kept git evidence optional and conservative so offline/limited-history environments do not block deterministic map generation.
- Treated known summary/planning files as supporting evidence by default unless explicit execution-spec headings are present.

## Reconstruction Notes
