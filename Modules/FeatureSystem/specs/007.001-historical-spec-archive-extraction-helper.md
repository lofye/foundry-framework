# Execution Spec: 007.001-historical-spec-archive-extraction-helper

Before implementing 008 fully, add a helper for preparing the historical spec archive.

Goal:
Create a local extraction/prep workflow that scans a directory of messy historical .md/.txt files and extracts likely individual specs into normalized candidate folders.

Input:
_import/raw-historical-specs/

Output:
_import/historical-specs/
candidate-001/
spec.md
source.md
metadata.json
candidate-002/
spec.md
source.md
metadata.json

Requirements:
- Do not require perfect source formatting.
- Detect likely spec boundaries using headings/patterns such as:
- "Spec 12"
- "Spec 12A"
- "Spec 35D"
- "# Spec"
- "Execution Spec:"
- "Title"
- "Purpose"
- Preserve original extracted text in `source.md`.
- Write cleaned candidate spec text to `spec.md`.
- Write best-effort metadata to `metadata.json`:
- original_file
- candidate_id
- detected_spec_label
- detected_title
- suggested_slug
- suggested_module
- confidence: high|medium|low
- notes
- Do not import into `Modules/*` yet.
- Do not create implementation-log entries yet.
- Do not create reconstruction notes yet.
- This is a prep/extraction step only.

Add a command if appropriate:

php bin/foundry historical-specs:extract --source=_import/raw-historical-specs --target=_import/historical-specs --json

Also support:

php bin/foundry historical-specs:extract --source=... --target=... --dry-run --json

Determinism:
- stable file ordering
- stable candidate numbering
- stable JSON keys
- no timestamps
- no absolute local paths in JSON

Testing:
- one file containing multiple specs
- one file containing no specs
- weird headings like "Spec 35D-2"
- duplicate titles
- low-confidence extraction
- dry-run writes nothing
- apply writes deterministic files

Do not implement the full import into Modules yet. This helper prepares the archive for 008.
