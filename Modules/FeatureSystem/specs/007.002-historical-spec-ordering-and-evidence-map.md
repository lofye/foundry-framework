# Execution Spec: 007.002-historical-spec-ordering-and-evidence-map

## Purpose

Add a deterministic historical-spec ordering and evidence-mapping layer before historical specs are imported into canonical module locations.

This spec builds on:

```text
Modules/FeatureSystem/specs/007.001-historical-spec-archive-extraction-helper.md
```

and prepares for:

```text
Modules/FeatureSystem/specs/008-import-historical-spec-archive.md
```

The archive extraction helper can split messy historical files into candidates, but the import workflow still needs durable rules for historical ordering, legacy spec labels, likely module mapping, evidence sources, confidence levels, git commit/diff cross-referencing, and files containing multiple specs with optional RESULT/OUTPUT blocks.

---

## Core Principle

Historical imports must be ordered and attributed from evidence, not guessed silently.

When evidence is incomplete, the system must preserve uncertainty explicitly.

---

## Background

The current historical archive contains many files under:

```text
_import/historical-specs/
```

Known historical facts:

- historical specs were generally implemented in alphabetical plus numerical order
- commit messages are often specific and can help reconstruct implementation order
- git diffs/revisions can help map specs to actual code changes
- numeric canonical IDs appear to begin around `Spec35D7C`, corresponding to `ContextPersistence/011-auto-planning-from-canonical-feature-context`
- `Foundry-Spec-Summaries.md` contains useful information about roughly the first 20 specs
- `Foundry-Plans-for-19A-19F.md` contains planning information for specs 19A through 19F
- some archive files contain multiple specs
- some files also contain optional `OUTPUT` or `RESULT` sections from Codex runs

This information should become machine-readable import context.

---

## Goals

1. Create a deterministic evidence map for historical specs.
2. Preserve known ordering hints.
3. Preserve source-file provenance.
4. Support multi-spec files.
5. Support optional RESULT/OUTPUT extraction.
6. Support git commit/diff cross-referencing.
7. Produce candidate metadata that `008-import-historical-spec-archive.md` can consume.
8. Avoid pretending uncertain mappings are certain.

---

## Non-Goals

- Do not import specs into `Modules/*` yet.
- Do not create module context files yet.
- Do not create reconstruction notes for historical specs yet.
- Do not append implementation-log entries yet.
- Do not rewrite historical source files.
- Do not require perfect mapping before import.
- Do not compact or summarize decision ledgers.

---

## New Concept: Historical Evidence Map

Introduce a generated evidence map file:

```text
_import/historical-specs/evidence-map.json
```

and optionally a human-readable report:

```text
_import/historical-specs/evidence-map.md
```

These files describe all extracted historical candidates, ordering, provenance, confidence, and supporting evidence.

---

## Evidence Map JSON Shape

The JSON must be deterministic and stable.

Suggested top-level shape:

```json
{
  "version": 1,
  "source_root": "_import/historical-specs",
  "ordering_strategy": "legacy_label_then_filename_then_candidate",
  "candidates": [
    {
      "candidate_id": "candidate-001",
      "source_file": "Foundry-Spec-35D7C.md",
      "legacy_label": "Spec35D7C",
      "legacy_order_key": "035.D.007.C",
      "detected_title": "Auto Planning From Canonical Feature Context",
      "suggested_module": "ContextPersistence",
      "suggested_spec_path": "Modules/ContextPersistence/specs/011-auto-planning-from-canonical-feature-context.md",
      "implemented": true,
      "confidence": "high",
      "evidence": {
        "source_text": "confirmed",
        "codex_result": "confirmed",
        "followups": "unknown",
        "git_commit": "inferred",
        "current_source": "inferred"
      },
      "notes": [
        "Canonical numeric IDs appear to begin at this point in the historical archive."
      ]
    }
  ]
}
```

Required candidate fields:

```text
candidate_id
source_file
legacy_label
legacy_order_key
detected_title
suggested_module
suggested_spec_path
implemented
confidence
evidence
notes
```

Allowed confidence values:

```text
high
medium
low
unknown
```

Allowed evidence values:

```text
confirmed
inferred
unknown
conflict
```

## Legacy Ordering Rules

Historical order must be deterministic.

The importer should parse labels such as:

```text
Spec 1
Spec 19A
Spec 19FB
Spec 30C-2
Spec 35D7C
Spec 35D7JA
Spec 35D7K
```

and convert them into stable sortable order keys.

Examples:

```text
Spec 1       -> 001
Spec 19A     -> 019.A
Spec 19FB    -> 019.F.B
Spec 30C-2   -> 030.C.002
Spec 35D7C   -> 035.D.007.C
Spec 35D7JA  -> 035.D.007.J.A
Spec 35D7K   -> 035.D.007.K
```

Rules:

- numeric parts sort numerically
- alpha parts sort alphabetically
- mixed labels preserve segment order
- hyphenated numeric suffixes sort as numeric segments
- unknown labels sort after known labels by filename
- stable filename fallback required

---

## Known Anchor: Numeric Canonical IDs

The evidence map must support an anchor rule:

```text
Spec35D7C ~= ContextPersistence/011-auto-planning-from-canonical-feature-context
```

This is a historical anchor, not a universal rule.

Requirements:

- represent this as evidence, not hidden code
- allow future anchors to be added
- do not infer all mappings from this anchor blindly
- mark anchor-derived mappings as `inferred` unless independently confirmed

Suggested optional config:

```text
_import/historical-specs/import-anchors.json
```

Example:

```json
{
  "anchors": [
    {
      "legacy_label": "Spec35D7C",
      "canonical_module": "ContextPersistence",
      "canonical_spec_id": "011",
      "canonical_slug": "auto-planning-from-canonical-feature-context",
      "confidence": "high",
      "notes": "Known point where numeric canonical IDs appear to begin."
    }
  ]
}
```

---

## Source Summary Files

The evidence mapper must recognize archive summary/planning files as evidence sources rather than direct specs unless they contain extractable specs.

Special files include:

```text
Foundry-Spec-Summaries.md
Foundry-Plans-for-19A-19F.md
```

Rules:

- they may inform metadata and notes
- they may produce candidates if they contain full specs
- otherwise they should be listed as supporting evidence files
- do not import them as ordinary specs by default

---

## Multi-Spec File Handling

Some files may contain multiple specs and optional result/output blocks.

The evidence map must support multiple candidates from one source file:

```json
{
  "source_file": "Foundry-Spec-35D7G-015.md",
  "source_segment": 1,
  "source_segments_total": 2,
  "result_detected": true,
  "result_file": "result.md"
}
```

If extraction writes candidate folders, optional files may include:

```text
candidate-041/spec.md
candidate-041/source.md
candidate-041/result.md
candidate-041/followups.md
candidate-041/metadata.json
```

## Git Evidence Cross-Reference

Add optional support for git evidence analysis.

Possible command:

```bash
php bin/foundry historical-specs:evidence --source=_import/historical-specs --json
```

or extend the extraction command with:

```bash
php bin/foundry historical-specs:extract --source=_import/raw-historical-specs --target=_import/historical-specs --with-git-evidence --json
```

Git evidence should inspect commit messages and optionally touched files.

Requirements:

- do not require internet access
- do not mutate git history
- do not depend on current branch name
- use deterministic ordering
- avoid absolute paths in JSON
- if git evidence unavailable, mark it `unknown`

Suggested evidence fields:

```json
{
  "git": {
    "matched_commits": [
      {
        "hash": "abc1234",
        "subject": "Implement Spec 35D7C auto planning",
        "confidence": "medium"
      }
    ]
  }
}
```

Commit hashes may be shortened deterministically.

## Module Suggestion Rules

The evidence mapper may suggest modules from:

- filename
- title
- known anchors
- current module names
- commit touched files
- existing source paths
- summary file descriptions

Examples:

```text
ContextPersistence
FeatureSystem
GenerateEngine
McpServer
Marketplace
StateStore
ExtensionSystem
QualityEnforcement
CompilerDeterminism
CliExperience
```

Rules:

- suggestions must include confidence
- low-confidence mappings must remain reviewable
- never silently import low-confidence candidates into active specs without review

## Commands

If implemented as CLI commands, add:

```bash
php bin/foundry historical-specs:evidence --source=_import/historical-specs --json
```

Optional flags:

```text
--anchors=_import/historical-specs/import-anchors.json
--with-git
--write
--dry-run
```

Behavior:

- `--dry-run`: report only
- `--write`: write `evidence-map.json` and optional markdown report
- default should be safe and non-mutating unless existing command conventions differ
