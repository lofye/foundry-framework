# 007.003-tighten-historical-import-boundary-and-module-inference

## Purpose

Tighten the historical spec extraction/evidence workflow after discovering a more accurate canonical-spec transition point.

This spec builds on:

```text
Modules/FeatureSystem/specs/007.001-historical-spec-archive-extraction-helper.md
Modules/FeatureSystem/specs/007.002-historical-spec-ordering-and-evidence-map.md
```

and prepares for:

```text
Modules/FeatureSystem/specs/008-import-historical-spec-archive.md
```

The key correction is:

```text
Spec35D1 == docs/features/context-persistence/001-initial.md
```

not the previously assumed `Spec35D7C` anchor.

That means the historical import boundary should distinguish:

```text
Pre-canonical era:
  Spec 0A through Spec 35D

Canonical numeric/spec-file era:
  Spec 35D1 and later
```

This spec also tightens multi-spec file handling and module inference for the pre-canonical era, where specs were not yet organized by Features or Modules.

---

## Core Principle

Historical reconstruction must preserve what is known, mark what is inferred, and avoid silently importing already-canonical specs.

When the source archive contains mixed eras, the tooling must identify the transition boundary before import.

---

## Background

After implementing `007.002`, the evidence mapper can parse legacy labels, produce order keys, handle summary/planning files, detect RESULT/FOLLOWUPS blocks, and optionally inspect git evidence.

Additional historical facts are now known:

1. The first canonical numeric spec was:
   ```text
   Spec35D1 == docs/features/context-persistence/001-initial.md
   ```

2. Therefore, all specs from `35D1` onward are part of the canonical spec-file era and should generally not be imported as pre-canonical historical specs.

3. Some archive files contain multiple specs, so canonical ordering and identity cannot be inferred from the filename alone.

4. Before `35D1`, specs were not divided into Foundry `Features` or `Modules`, so module assignment is inherently interpretive and must be evidence-scored.

---

## Goals

1. Add an explicit historical-era classifier.
2. Treat `Spec35D1` as the canonical transition anchor.
3. Prevent accidental re-import of already-canonical `35D1+` specs as historical pre-canonical specs.
4. Preserve candidate-level handling for multi-spec files.
5. Add evidence-based module inference for pre-canonical specs.
6. Support reviewable low-confidence module mappings.
7. Keep `008-import-historical-spec-archive.md` from importing candidates that should instead be linked to existing canonical specs.
8. Preserve uncertainty explicitly in evidence maps.

---

## Non-Goals

- Do not import specs into `Modules/*` yet.
- Do not delete archive files.
- Do not move files automatically out of `_import/historical-specs/`.
- Do not create module context docs yet.
- Do not create historical reconstruction notes yet.
- Do not append implementation-log entries yet.
- Do not assume filename equals spec identity.
- Do not force a single module assignment when evidence is weak.
- Do not rewrite already-implemented `007.002`; extend/tighten it.

## New Concept: Historical Era

Each candidate must be classified into one of:

```text
pre_canonical
canonical_existing
ambiguous
supporting_evidence
```

### `pre_canonical`

A candidate that appears to belong before `Spec35D1`.

These are candidates for import by `008`.

Examples:

```text
Spec 0A
Spec 1
Spec 19A
Spec 30C-2
Spec 35D
```

### `canonical_existing`

A candidate at or after `Spec35D1`.

These should not be imported as new historical specs by default. They should be linked to existing canonical specs when possible.

Examples:

```text
Spec35D1
Spec35D2
Spec35D7C
Spec35D7K
Spec36
Spec44
```

### `ambiguous`

A candidate whose label or evidence is insufficient to decide.

Ambiguous candidates must not be imported into active module specs without review.

### `supporting_evidence`

A file or segment that informs mapping but is not itself an importable spec.

Examples:

```text
Foundry-Spec-Summaries.md
Foundry-Plans-for-19A-19F.md
```

## Canonical Transition Anchor

The evidence mapper must represent this anchor explicitly:

```json
{
  "legacy_label": "Spec35D1",
  "canonical_path": "docs/features/context-persistence/001-initial.md",
  "canonical_module": "ContextPersistence",
  "canonical_spec_id": "001",
  "canonical_slug": "initial",
  "era": "canonical_existing",
  "confidence": "high",
  "notes": [
    "First known canonical numeric spec-file era anchor."
  ]
}
```

If `docs/features/context-persistence/001-initial.md` has since moved or been migrated to a module path, the mapper may also include the current path as an additional field:

```json
{
  "current_canonical_path": "Modules/ContextPersistence/specs/001-initial.md"
}
```

Only include this if discoverable from current repo state or explicit anchor configuration.

## Anchor Configuration

Extend anchor support so `_import/historical-specs/import-anchors.json` can represent era boundaries.

Suggested shape:

```json
{
  "canonical_transition": {
    "legacy_label": "Spec35D1",
    "canonical_path": "docs/features/context-persistence/001-initial.md",
    "canonical_module": "ContextPersistence",
    "canonical_spec_id": "001",
    "canonical_slug": "initial",
    "confidence": "high"
  },
  "anchors": [
    {
      "legacy_label": "Spec35D1",
      "canonical_module": "ContextPersistence",
      "canonical_spec_id": "001",
      "canonical_slug": "initial",
      "confidence": "high"
    }
  ]
}
```

Rules:

- transition anchor must be explicit, not hardcoded silently.
- if no transition anchor file exists, default behavior may use the built-in known anchor from this spec.
- user-provided anchor config overrides built-in defaults.
- anchor-derived classifications must be visible in JSON evidence.

## Candidate Metadata Additions

Each candidate in `evidence-map.json` must include:

```json
{
  "era": "pre_canonical|canonical_existing|ambiguous|supporting_evidence",
  "import_action": "import|link_existing|review|ignore_supporting",
  "canonical_transition_relative": "before|at|after|unknown",
  "module_inference": {
    "suggested_module": "ContextPersistence",
    "confidence": "medium",
    "evidence": [
      {
        "type": "title_keyword",
        "value": "context persistence",
        "confidence": "medium"
      }
    ],
    "alternatives": [
      {
        "module": "FeatureSystem",
        "confidence": "low",
        "reason": "Spec affects spec workflow."
      }
    ]
  }
}
```

Allowed `import_action` values:

```text
import
link_existing
review
ignore_supporting
```

Rules:

- `pre_canonical` + sufficient module confidence -> `import`
- `canonical_existing` -> `link_existing`
- `ambiguous` -> `review`
- `supporting_evidence` -> `ignore_supporting`

## Multi-Spec File Handling

Do not infer candidate identity only from the containing filename.

If one file contains multiple specs, each segment must be independently classified.

Example:

```json
{
  "source_file": "Foundry-Spec-35D7G-015.md",
  "source_segment": 2,
  "legacy_label": "Spec35D7G2",
  "era": "canonical_existing",
  "import_action": "link_existing"
}
```

If the filename suggests one label but the internal segment heading suggests another, prefer the internal segment heading and record the filename mismatch as evidence:

```json
{
  "notes": [
    "Filename label differs from internal segment label; internal heading used for candidate identity."
  ]
}
```

## Module Inference For Pre-Canonical Specs

Pre-35D1 specs were not organized by Features or Modules.

Therefore module assignment must be inferred from evidence.

Evidence sources may include:

- title keywords
- Purpose section
- Goals section
- files mentioned in spec
- Codex RESULT/OUTPUT touched files
- git commit subject/body
- git diff touched paths
- current source files
- `Foundry-Spec-Summaries.md`
- `Foundry-Plans-for-19A-19F.md`

Suggested module inference signals:

```text
src/Context/*                     -> ContextPersistence or FeatureSystem depending content
src/Generate/*                    -> GenerateEngine
src/MCP/*                         -> McpServer
src/Marketplace/*                 -> Marketplace
src/Quality/*                     -> QualityEnforcement
src/CLI/*                         -> CliExperience unless command belongs clearly to another module
docs/features/* or spec workflow  -> FeatureSystem / ContextPersistence
compiler/graph/verify contracts   -> CompilerDeterminism
examples / thresholds             -> Examples / Thresholds / website-related, mark review if module absent
```

Do not create new modules automatically unless explicitly required by later import review.

If no confident module exists, set:

```json
{
  "suggested_module": null,
  "confidence": "low",
  "import_action": "review"
}
```

---

## Git Evidence

If `--with-git` is used, evidence mapper should cross-reference:

- commit subjects containing the legacy label
- commit bodies containing the legacy label
- commits near the likely order
- touched paths that suggest module assignment

If a commit matches a canonical-existing candidate, include it as linking evidence rather than import evidence.

If git evidence conflicts with filename/title inference, mark:

```text
conflict
```

and set import action to `review`.

---

## Canonical Existing Candidate Handling

Candidates from `Spec35D1` onward should be treated as already-canonical unless evidence proves otherwise.

For these candidates, evidence map should attempt to produce:

```json
{
  "era": "canonical_existing",
  "import_action": "link_existing",
  "existing_spec_path": "Modules/ContextPersistence/specs/001-initial.md",
  "confidence": "medium"
}
```

If existing spec path cannot be found:

```json
{
  "era": "canonical_existing",
  "import_action": "review",
  "notes": [
    "Candidate appears to be canonical-era but no matching current spec path was found."
  ]
}
```
