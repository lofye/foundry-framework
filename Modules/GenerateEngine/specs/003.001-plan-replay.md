# Execution Spec: 003.001-plan-replay

## Feature
- generate-engine

## Purpose
- Re-execute a previously persisted generate plan safely and deterministically.
- Make past generate runs reusable rather than one-time artifacts.
- Establish replay behavior that is explicit, validated, and trustworthy.

## Scope
- Add replay commands for persisted plans.
- Support strict replay and adaptive replay modes.
- Reuse persisted plan artifacts created by plan persistence.
- Revalidate plans before execution.
- Keep this spec focused on replay, not undo.

## Constraints
- Replay must be explicit and never automatic.
- Replay must reuse persisted plan data rather than regenerating a new plan silently.
- Replay must run validation before execution.
- Replay must fail clearly when reproducibility or safety requirements are not met.
- Do not silently diverge from the stored plan.
- Reuse existing `GenerationPlan`, `PlanValidator`, verification, and safety-routing infrastructure where practical.
- Do not implement full snapshot-based rollback in this spec.

## Inputs

Expect inputs such as:
- persisted plan artifacts under `.foundry/plans/`
- stored original/final plan data
- execution metadata
- current repository/environment state

If any critical input is missing:
- fail clearly and deterministically
- do not invent substitute plan data
- do not silently regenerate a new plan and call it replay

## Requested Changes

### 1. Add `plan:replay <plan_id>`

Add:

```bash
foundry plan:replay <plan_id>
```

This command must:
- load the persisted plan artifact
- select the correct replay source plan (`plan_final` when present and approved, otherwise the original executable plan)
- revalidate before execution
- execute only if replay remains valid and safe

### 2. Support Replay Modes

#### Adaptive Replay (default)

Default replay may:
- revalidate the stored plan against current conditions
- proceed when differences are judged safe by existing validators/routing
- warn clearly about drift when it exists

Adaptive replay must not silently mutate the persisted plan artifact itself.

#### Strict Replay

Add:

```bash
foundry plan:replay <plan_id> --strict
```

Strict replay must:
- require materially matching replay conditions
- fail if significant drift or incompatibility is detected
- never silently adapt around broken assumptions

### 3. Add Dry-Run Support

Replay must support a dry-run mode.

Add support for:

```bash
foundry plan:replay <plan_id> --dry-run
```

Dry-run must:
- load the persisted plan
- validate replayability
- show what would execute
- make no file changes

### 4. Reuse Existing Validation and Safety Paths

Replay must reuse, where practical:
- stored `GenerationPlan`
- `PlanValidator`
- verification runner
- generate safety/risk routing

Do not create a second disconnected execution pathway.

### 5. Surface Drift Explicitly

If current conditions differ materially from the stored replay assumptions:
- report the drift clearly
- distinguish strict failure from adaptive warning
- do not hide the divergence

Acceptable drift reporting may include:
- changed repository state
- missing target files
- incompatible mode/target assumptions
- validator failures

### 6. Output Contract

Human output should include:
- plan id
- replay mode
- replayability result
- warnings or drift notices
- execution result
- verification result

JSON output should include, at minimum:
- `plan_id`
- `replay_mode`
- `status`
- `drift_detected`
- `drift_summary`
- `actions_executed`
- `verification`
- `dry_run`

### 7. Keep Replay Deterministic

Replay must:
- use the stored plan as the execution basis
- execute in deterministic order
- fail clearly when deterministic replay is not possible

No silent divergence.

### 8. Tests

Add focused coverage proving:

- persisted plans can be replayed by id
- strict replay fails on material drift
- adaptive replay surfaces warnings and only proceeds when safe
- dry-run replay makes no changes
- replay uses stored plan artifacts rather than silently generating a new plan
- output is deterministic and machine-readable
- all relevant generate tests still pass

## Non-Goals
- Do not implement undo in this spec.
- Do not implement team approval workflows.
- Do not add patch-based rollback.
- Do not require external storage.
- Do not silently auto-heal incompatible plans.

## Authority Rule
- Replay must be explicit, validated, and based on persisted plan artifacts.
- Strict replay must fail clearly when reproducibility is not satisfied.
- Adaptive replay may proceed only when validators and safety rules allow it.

## Completion Signals
- `plan:replay` exists
- strict replay exists
- dry-run replay exists
- persisted plans can be replayed safely and deterministically
- drift is surfaced clearly
- all tests pass

## Post-Execution Expectations
- Foundry can reuse past generate runs as controlled, trustworthy inputs.
- Replay becomes a real operational tool rather than a thought experiment.
- Later collaboration and policy layers can build on persisted replayable plans.
