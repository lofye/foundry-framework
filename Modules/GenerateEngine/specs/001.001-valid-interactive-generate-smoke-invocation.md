# Execution Spec: 001.001-valid-interactive-generate-smoke-invocation

## Feature
- generate-engine

## Purpose
- Add a valid, deterministic smoke invocation for interactive generate so the repository verifies the interactive entry path with a real supported command shape.
- Replace any invalid or misleading smoke usage that fails before interactive review begins.
- Ensure `generate --interactive` is exercised through a harmless review-only path.

## Scope
- Add or update a repository-owned smoke/integration path that invokes:
  - `foundry generate "<intent>" --mode=<new|modify|repair> --interactive`
- Ensure the invocation reaches the interactive review layer instead of failing early on missing required flags.
- Keep this focused on valid invocation coverage and harmless entry-path verification.
- Do not redesign interactive generate behavior in this spec.

## Constraints
- The smoke invocation must be valid under the actual CLI contract.
- The invocation must be deterministic and safe.
- Prefer a harmless plan/review path that does not depend on fragile repository state.
- Do not require destructive execution for the smoke check.
- Do not weaken existing generate validation rules.
- Reuse existing interactive generate infrastructure and tests where practical.

## Inputs

Expect inputs such as:
- `foundry generate`
- `--interactive`
- required `--mode=new|modify|repair`
- current interactive review flow
- CLI integration test infrastructure

If any critical input is missing:
- fail clearly and deterministically
- do not invent unsupported flags or command shapes

## Requested Changes

### 1. Use a Valid Interactive Generate Invocation

Add or update the repository-owned smoke/integration coverage so it uses a valid command shape such as:

```bash
foundry generate "<intent>" --mode=new --interactive --json
```

The exact intent string may vary, but it must be:
- harmless
- deterministic
- suitable for exercising interactive plan review

### 2. Ensure the Invocation Reaches Review Logic

The smoke path must prove more than argument validation.

It must reach the interactive review layer and exercise at least:
- plan creation
- review rendering or review result construction
- interactive decision handling

A failure like `GENERATE_MODE_REQUIRED` must no longer be the observed outcome of the smoke path.

### 3. Keep the Smoke Path Non-Destructive

The smoke path should prefer a review-only or rejected execution path so it does not rely on risky filesystem mutation.

Acceptable patterns include:
- rejecting the reviewed plan
- simulating user rejection
- using a no-op or harmless preview-oriented path if already supported

The goal is to verify valid interactive entry, not to perform risky generation in a smoke test.

### 4. Keep Output Deterministic

If JSON output is used, the smoke path should assert stable, machine-readable fields that prove interactive mode was entered successfully.

Prefer assertions on:
- status/result shape
- review/plan presence
- recorded user decision
- absence of early validation failure

Avoid assertions that depend on unstable formatting unless such formatting is already canonical.

### 5. Update Any Misleading Docs or Examples

If repository docs, checklists, or examples currently use an invalid interactive generate invocation, update them to the valid command shape.

Keep this minimal and only where necessary.

### 6. Tests

Add or update focused coverage proving:

- interactive generate is invoked with a valid `--mode`
- the interactive review layer is actually reached
- the smoke path does not fail with `GENERATE_MODE_REQUIRED`
- the chosen path remains non-destructive
- output remains deterministic
- all related generate tests still pass

## Non-Goals
- Do not redesign `generate --interactive`.
- Do not add new interactive features in this spec.
- Do not broaden this into full workflow testing of every interactive branch.
- Do not weaken CLI validation requirements.
- Do not depend on a graphical UI.
