# Execution Spec: 003.002-plan-undo

## Feature
- generate-engine

## Purpose
- Provide a safe, explicit, best-effort undo mechanism for persisted generate plans.
- Allow developers to reverse certain applied generate changes with clear visibility into what is and is not reversible.
- Make undo trustworthy by being conservative, explicit, and limited rather than pretending to offer perfect rollback.

## Scope
- Add an undo command for persisted plans.
- Support best-effort reversal for reversible file actions.
- Surface irreversible and partially reversible cases clearly.
- Support dry-run undo previews.
- Keep this spec focused on conservative V1 undo behavior.

## Constraints
- Undo must be explicit and never automatic.
- Undo must be best-effort, conservative, and honest about limitations.
- Undo must not silently claim full reversibility when only partial reversal is possible.
- Reversible behavior must rely on data already persisted or safely derivable.
- Do not require full repository snapshots in this spec.
- Do not require git integration in this spec.
- Do not perform destructive undo without explicit confirmation where appropriate.

## Inputs

Expect inputs such as:
- persisted plan artifacts under `.foundry/plans/`
- stored affected files / actions executed
- pre-change file contents when available
- current filesystem state

If any critical input is missing:
- fail clearly and deterministically
- do not invent rollback data
- do not report success for actions that cannot actually be reversed

## Requested Changes

### 1. Add `plan:undo <plan_id>`

Add:

```bash
foundry plan:undo <plan_id>
```

This command must:
- load the persisted plan artifact
- determine the reversible action set
- present undoability clearly
- execute only after explicit confirmation when destructive reversal is involved

### 2. Define V1 Undo Strategy

Use a conservative best-effort action mapping:

- `create_file` → delete created file when safe
- `update_file` → restore previous contents only if previous contents were persisted or safely available by explicit contract
- `delete_file` → not reversible in V1 unless previous contents were persisted and available
- non-file or ambiguous actions → not automatically reversible in V1 unless explicit deterministic reversal logic exists

Do not pretend an irreversible action is reversible.

### 3. Persist or Require Reversal Inputs Explicitly

If V1 undo for updated files requires previous contents, the persisted plan record or related persisted data must explicitly store what is needed.

This spec may extend the persisted plan artifact from `003` to include the minimal rollback inputs necessary for the reversible V1 action set.

That extension must remain deterministic and local.

### 4. Add Dry-Run Support

Undo must support:

```bash
foundry plan:undo <plan_id> --dry-run
```

Dry-run must:
- show what would be reversed
- show what cannot be reversed
- make no file changes

### 5. Surface Partial Undo Honestly

Undo output must clearly distinguish:
- fully reversible actions
- partially reversible actions
- irreversible actions
- actions skipped because conditions are unsafe or missing

If undo is partial, the result must say so explicitly.

### 6. Safeguards

Undo must enforce:
- no automatic undo
- explicit user intent
- warnings for irreversible or partial outcomes
- extra confirmation when destructive file deletion is part of the reversal
- refusal when current state makes reversal clearly unsafe

### 7. Output Contract

Human output should include:
- plan id
- undoability summary
- reversible actions
- irreversible actions
- dry-run or live execution result
- partial-warning status

JSON output should include, at minimum:
- `plan_id`
- `status`
- `dry_run`
- `fully_reversible`
- `reversed_actions`
- `irreversible_actions`
- `skipped_actions`
- `warnings`

### 8. Keep Undo Deterministic

Undo planning and execution must be deterministic with respect to:
- persisted artifact inputs
- current filesystem state
- explicit safeguards

No silent best guesses.

### 9. Tests

Add focused coverage proving:

- undo by plan id works for reversible file-creation cases
- update-file undo only works when prior contents are explicitly available
- irreversible actions are reported honestly
- dry-run undo makes no changes
- partial undo warnings are emitted correctly
- destructive reversal is gated appropriately
- all relevant generate tests still pass

## Non-Goals
- Do not implement perfect rollback for all action types.
- Do not add git-based rollback in this spec.
- Do not add collaborative approval workflows.
- Do not add snapshot/patch history beyond what is minimally needed for V1 reversible actions.
- Do not silently reconstruct deleted content from guesswork.

## Authority Rule
- Undo must be explicit, conservative, and honest about reversibility.
- Partial undo is acceptable in V1; pretending partial undo is full undo is not.
- Reversal must be based on persisted or explicitly available data, not guesswork.

## Completion Signals
- `plan:undo` exists
- dry-run undo exists
- reversible actions can be reversed safely
- irreversible and partial cases are reported clearly
- persisted plan data is sufficient for the supported V1 reversible cases
- all tests pass

## Post-Execution Expectations
- Foundry gains a trustworthy first undo layer for generate operations.
- Developers can safely reverse a meaningful subset of generate changes.
- The system is prepared for later richer rollback support without overstating V1 capabilities.
