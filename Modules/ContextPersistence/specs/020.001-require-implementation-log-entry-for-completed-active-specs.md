# Execution Spec: 020.001-require-implementation-log-entry-for-completed-active-specs

## Feature
- context-persistence

## Purpose
- Make implementation-log coverage an enforced invariant rather than a documentation convention.
- Ensure completed active execution specs always have a corresponding entry in `docs/features/implementation-log.md`.
- Prevent verification from passing when spec implementation history is incomplete or silently missing.

## Scope
- Add verification for implementation-log coverage of completed active execution specs.
- Define deterministic matching rules between active spec files and implementation-log entries.
- Fail clearly when a completed active spec is missing a required log entry.
- Keep this focused on verification and enforcement, not on redesigning the execution lifecycle.

## Constraints
- Keep verification deterministic and machine-readable.
- Do not rely on LLM memory or prompt wording to enforce logging.
- Do not require draft specs to appear in the implementation log.
- Do not treat future/unimplemented active specs as completed solely because they exist.
- Reuse existing spec-validation and context-verification infrastructure where practical.
- Prefer the smallest enforceable rule set over a broad historical-audit system.

## Inputs

Expect inputs such as:
- active execution specs under `docs/features/<feature>/specs/`
- draft execution specs under `docs/features/<feature>/specs/drafts/`
- `docs/features/implementation-log.md`
- implementation state implied by already-promoted active specs

If any critical input is missing:
- fail clearly and deterministically
- do not invent log entries
- do not silently ignore missing implementation history

## Requested Changes

### 1. Define the Enforcement Rule

A completed active execution spec must have a matching entry in:

```text
docs/features/implementation-log.md
```

For this spec, treat every active execution spec under:

```text
docs/features/<feature>/specs/<id>-<slug>.md
```

as requiring a log entry unless the system already has a more explicit way to distinguish “active but not yet implemented.”

If such an explicit distinction already exists, use it.
If not, prefer the current practical rule:
- active spec = requires implementation-log coverage
- draft spec = does not require implementation-log coverage

### 2. Define Matching Rules

Matching must be deterministic.

At minimum, a log entry must match the canonical spec path in this form:

```md
- spec: <feature>/<id>-<slug>.md
```

The verifier must:
- normalize path matching consistently
- match exact active spec identity
- reject missing entries
- ignore draft specs

Do not use fuzzy matching.

### 3. Add a Verification Surface

Add enforcement to an existing verification path so missing log entries cause verification failure.

Acceptable homes include:
- `spec:validate`
- a dedicated implementation-log verification step used by existing verification flows
- another existing deterministic verification surface

Choose the smallest place that makes the rule real and easy to run routinely.

### 4. Emit Stable Violations

If a required entry is missing, emit a stable violation/error with:
- a deterministic code
- a clear message
- the relevant spec path

Example direction:
- `EXECUTION_SPEC_IMPLEMENTATION_LOG_MISSING`

The exact code may vary, but it must be explicit and stable.

### 5. Keep Drafts Out of Scope

Do not require log entries for:
- specs under `docs/features/<feature>/specs/drafts/`

Drafts remain planning artifacts only.

### 6. Keep Existing Log Format Canonical

Do not redesign the log format in this spec.

Continue using the established format in `docs/features/implementation-log.md`, including canonical spec path references.

### 7. Tests

Add focused coverage proving:

- active specs with matching log entries pass
- active specs missing log entries fail
- draft specs do not require log entries
- matching is exact and deterministic
- repeated verification runs are stable
- all relevant existing verification tests still pass

## Non-Goals
- Do not auto-generate missing log entries in this spec.
- Do not redesign planner or execution lifecycles in this spec.
- Do not add fuzzy historical inference.
- Do not require decision-ledger entries for every spec in this spec.
- Do not broaden this into a general repository audit beyond implementation-log enforcement.

## Canonical Context
- Canonical feature spec: `docs/context-persistence/context-persistence.spec.md`
- Canonical feature state: `docs/context-persistence/context-persistence.md`
- Canonical decision ledger: `docs/context-persistence/context-persistence.decisions.md`

## Authority Rule
- Implementation-log coverage for completed active specs is mandatory.
- Draft specs remain exempt.
- Verification must fail deterministically when required implementation history is missing.

## Completion Signals
- Missing implementation-log entries for active specs are detectable and fail verification.
- Draft specs remain exempt.
- Matching is exact and deterministic.
- Existing verification surfaces remain usable.
- All tests pass.

## Post-Execution Expectations
- Codex and humans can no longer “forget” implementation-log entries without verification catching it.
- Implementation history becomes more trustworthy and complete.
- The spec lifecycle becomes stricter: active implemented work must leave an auditable log trail.
