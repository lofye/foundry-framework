# Execution Spec: 015.001-context-doctor-execution-spec-drift

## Feature
- context-persistence

## Purpose
- Make `foundry context doctor` detect when execution specs exist for a feature but canonical feature context is missing or incomplete.
- Prevent developers and agents from treating execution specs as authoritative in the absence of canonical feature context.
- Fit the new drift check into the current `context doctor` and `verify context` output contracts without introducing a second competing diagnostics shape.

## Scope
- Add a new feature-scoped context diagnostic for execution-spec drift.
- Detect execution-spec presence under `docs/features/<feature>/specs/` and `docs/features/<feature>/specs/drafts/`.
- Detect missing canonical context files under `docs/features/`.
- Surface this condition through the existing `context doctor` file-bucket issue model.
- Let `verify context` surface the same doctor issue through its existing flattened `issues` list.
- Keep this focused on detection and reporting only.

## Constraints
- Keep canonical feature context authoritative.
- Keep execution specs non-authoritative.
- Keep output deterministic and machine-readable.
- Reuse existing doctor and verify-context plumbing where practical.
- Do not auto-create, rewrite, promote, or delete any files.
- Do not infer canonical context from execution specs.
- Do not broaden this into execution-spec naming, allocation, promotion, or validation behavior.
- Do not introduce a new top-level diagnostics collection for `context doctor`.

## Requested Changes

### 1. Add a New Doctor Issue Code

Add a new context-doctor issue code:

`EXECUTION_SPEC_DRIFT`

This issue must be detected when both of the following are true for a feature:

- one or more execution specs exist under:
  - `docs/features/<feature>/specs/`
  - `docs/features/<feature>/specs/drafts/`
- one or more required canonical feature context files are missing:
  - `docs/features/<feature>/<feature>.spec.md`
  - `docs/features/<feature>/<feature>.md`
  - `docs/features/<feature>/<feature>.decisions.md`

This issue exists to signal that planning artifacts exist for a feature whose canonical context is missing or incomplete.

### 2. Detection Rules

Detection must be feature-scoped.

For the target feature:

1. Discover whether any execution specs exist in either:
   - the active feature spec directory
   - the draft feature spec directory

2. Compute the set of missing canonical feature context files.

3. Emit `EXECUTION_SPEC_DRIFT` only when:
   - execution specs exist
   - and the missing canonical file set is non-empty

Do not emit the issue when:
- no execution specs exist
- all canonical context files exist

### 3. Context Doctor Integration

`foundry context doctor --feature=<feature> --json` must continue using its current output shape:

- top-level `status`
- top-level `can_proceed`
- top-level `requires_repair`
- `files.spec`, `files.state`, and `files.decisions`
- per-file `issues`
- top-level `required_actions`

Do not add a separate top-level diagnostics list in this spec.

Instead:
- append `EXECUTION_SPEC_DRIFT` to the `issues` array of each missing canonical file bucket that participates in the drift condition
- keep the standard issue shape already used by doctor:
  - `code`
  - `message`
  - `file_path`

Use a stable message equivalent to:

`Execution specs exist for this feature, but canonical feature context is missing or incomplete.`

Use the missing canonical file path as `file_path`.

### 4. Verify-Context Integration

`foundry verify context --feature=<feature> --json` must continue using its current output shape:

- top-level `status`
- top-level `can_proceed`
- top-level `requires_repair`
- flat `issues` list
- top-level `required_actions`

The new doctor issue must flow into `verify context` through the existing doctor-to-verify path.

In `verify context`, the surfaced issue must include:
- `source = doctor`
- `code = EXECUTION_SPEC_DRIFT`
- the same deterministic `message`
- the relevant `file_path`

### 5. Repair Semantics

For this implementation:
- treat `EXECUTION_SPEC_DRIFT` as a repair-required context problem
- keep `context doctor` and `verify context` conservative
- do not allow meaningful implementation to proceed while this condition exists

Recommended actions must include guidance equivalent to:
- create or initialize missing canonical context files
- run `foundry context init <feature> --json` when appropriate
- do not rely on execution specs as the source of truth

These actions should be appended through the existing `required_actions` model rather than a new side channel.

### 6. Deterministic Ordering

If multiple file issues are present:
- issue ordering must remain deterministic within each file bucket
- repeated runs against the same filesystem state must return the same ordered JSON
- verify-context flattened issue ordering must remain deterministic as well

If sorting is required, define it centrally and keep it stable.

### 7. Tests

Add focused coverage proving:

- no issue is emitted when canonical context exists and execution specs also exist
- no issue is emitted when canonical context is missing but no execution specs exist
- the issue is emitted when execution specs exist and canonical context is missing or incomplete
- active specs trigger the issue
- draft specs trigger the issue
- the issue is attached to the missing canonical file bucket(s)
- doctor JSON output preserves the current shape
- verify-context surfaces the issue through `source = doctor`
- required actions remain actionable and deterministic
- all existing related context doctor and verify-context tests still pass

## Non-Goals
- Do not add execution-spec auto-promotion.
- Do not generate canonical context from execution specs.
- Do not change execution-spec naming rules.
- Do not add UI-only behavior here.
- Do not redesign the current doctor/verify output contracts in this spec.
- Do not broaden this into a generic docs-health sweep outside feature-scoped context diagnostics.

## Canonical Context
- Canonical feature spec: `docs/context-persistence/context-persistence.spec.md`
- Canonical feature state: `docs/context-persistence/context-persistence.md`
- Canonical decision ledger: `docs/context-persistence/context-persistence.decisions.md`

## Authority Rule
- Canonical feature context remains the only authoritative source of feature intent and state.
- Execution specs remain planning artifacts only.
- When execution specs exist without complete canonical context, the system must diagnose that condition explicitly through the existing doctor/verify pipelines and require repair before meaningful implementation proceeds.

## Completion Signals
- `context doctor` emits `EXECUTION_SPEC_DRIFT` deterministically when execution specs exist without complete canonical context.
- The issue is attached through the existing per-file doctor issue model.
- `verify context` surfaces the same condition through its existing flattened doctor issue model.
- Missing canonical files are surfaced clearly.
- Required actions are actionable and deterministic.
- No false positives occur when canonical context is complete.
- All tests pass.

## Post-Execution Expectations
- Developers and agents get an explicit repair-required issue when execution specs exist without canonical feature context.
- The system more strongly enforces context anchoring discipline.
- Execution specs are less likely to be misused as a substitute for canonical feature context.
