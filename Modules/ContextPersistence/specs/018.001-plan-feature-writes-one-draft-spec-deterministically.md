# Execution Spec: 018.001-plan-feature-writes-one-draft-spec-deterministically

## Feature
- context-persistence

## Purpose
- Fix `plan feature` so one invocation writes at most one execution spec file.
- Ensure planner result contracts truthfully report the exact spec file written.
- Move planner-generated specs into `drafts/` so planning remains a proposal step rather than direct creation of active executable specs.

## Scope
- Harden the `plan feature` write path.
- Ensure planner-generated specs are created under `docs/features/<feature>/specs/drafts/`.
- Ensure the returned `spec_id`, `spec_path`, and `actions_taken` match the actual filesystem side effects.
- Keep this focused on planner file creation, lifecycle correctness, and result-contract integrity.

## Constraints
- Keep planning deterministic.
- Do not create more than one spec file for one planner invocation.
- Do not report a generated spec unless that exact file exists.
- Do not write planner-generated specs directly into the active spec directory.
- Do not redesign the full planner pipeline in this spec.
- Reuse existing allocation and resolution logic where practical.
- Prefer the smallest surgical fix over a broad refactor.

## Inputs

Expect inputs such as:
- `foundry plan feature <feature> --json`
- canonical feature context under `docs/features/`
- existing active and draft execution specs under `docs/features/<feature>/specs/`

If any critical input is missing:
- fail clearly and deterministically
- do not invent missing feature context
- do not silently skip write failures

## Requested Changes

### 1. Enforce Single-Write Planner Behavior

A single `plan feature` invocation must produce exactly one of these outcomes:

- write exactly one draft execution spec file
- return a clean blocked result
- fail clearly before reporting success

It must never create multiple spec files for one invocation.

### 2. Write Planner Output to Drafts Only

Planner-generated specs must be created at:

```text
docs/features/<feature>/specs/drafts/<id>-<slug>.md
```

Do not create planner-generated specs directly at:

```text
docs/features/<feature>/specs/<id>-<slug>.md
```

Planner output is a proposed work item and must remain non-executable until deliberately promoted out of `drafts/`.

### 3. Keep the Result Contract Truthful

If `plan feature --json` returns:

- `spec_id`
- `spec_path`
- `actions_taken: ["generated execution spec"]`

then:
- the exact file at `spec_path` must exist
- no additional planner-created spec files may have been written by that invocation

The result must describe the actual side effects, not an internal intention.

### 4. Fix Duplicate or Multi-Write Root Cause

Find and fix the exact cause of the observed behavior where one planner invocation wrote multiple spec files.

At minimum:
- identify where ids are allocated
- identify where planner output paths are built
- identify where file writes occur
- ensure the write path runs once per successful planner invocation

### 5. Preserve Deterministic ID Allocation

Planner allocation must remain deterministic.

Requirements:
- allocate only one id per invocation
- do not advance the next-id state more than once
- do not create hidden or intermediate spec files
- ensure the chosen id is the one reported in the result

### 6. Enforce Post-Write Verification

After the planner writes the file:
- verify the file exists at the exact `spec_path`
- verify only the intended file was created by that invocation

If post-write verification fails:
- do not report `planned`
- fail clearly and deterministically

### 7. Preserve Existing Planner Semantics Otherwise

This spec changes:
- file creation location
- single-write guarantee
- result-contract honesty

It does not change:
- bounded-step planning rules
- fallback-blocking behavior from 016
- generic planner refusal semantics
- execution-spec naming rules

### 8. Tests

Add focused coverage proving:

- one `plan feature` invocation creates at most one spec file
- planner-created specs are written to `drafts/`
- the returned `spec_id` matches the file actually written
- the returned `spec_path` matches the file actually written
- no hidden extra files are created
- duplicate or double allocation does not occur
- planner fails clearly if the write contract cannot be satisfied
- all relevant planner and CLI tests still pass

## Non-Goals
- Do not redesign the full planner architecture.
- Do not auto-promote drafts.
- Do not change execution-spec naming or validation rules.
- Do not broaden this into a general filesystem transaction framework.
- Do not change framework-repo execution blocking behavior for `implement spec`.

- Planner-generated specs land in `docs/features/<feature>/specs/drafts/`.
