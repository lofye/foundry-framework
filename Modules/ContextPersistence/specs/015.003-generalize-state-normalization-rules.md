# Execution Spec: 015.003-generalize-state-normalization-rules

## Feature
- context-persistence

## Purpose
- Promote the one-off state normalization work previously applied to `context-persistence` into a reusable, deterministic normalization system for feature state documents.
- Ensure state-document ordering and formatting are canonical across the repository.
- Reduce noisy diffs and prevent drift detection from reacting to ordering or formatting noise instead of real state changes.

## Scope
- Normalize `docs/features/<feature>/<feature>.md` state documents.
- Define canonical section ordering and content-shape rules for state documents.
- Introduce a reusable normalization path that can be called by framework code instead of relying on one-off manual cleanup.
- Keep this focused on feature state documents only for this implementation pass.

## Constraints
- Keep normalization deterministic and idempotent.
- Do not change the semantic meaning of a state document.
- Do not rewrite decision ledgers in this spec.
- Do not normalize execution specs in this spec.
- Do not introduce feature-specific normalization rules.
- Reuse existing context-document parsing/writing behavior where practical.
- Prefer a small, explicit normalization system over a broad formatting engine.

## Inputs

Expect inputs such as:
- a valid feature state document at `docs/features/<feature>/<feature>.md`
- state documents containing `Current State`, `Open Questions`, and `Next Steps`
- state documents whose bullets may be duplicated, out of order, stale, or inconsistently phrased

If any critical input is missing:
- fail clearly and deterministically
- do not invent missing state content
- do not auto-create unrelated context files in this spec

## Requested Changes

### 1. Introduce a Reusable State Normalization Path

Create a reusable normalization path for feature state documents.

This may be implemented as:
- a dedicated normalizer class
- a service used by context tooling
- or another deterministic internal abstraction

It must accept raw state-document content and produce canonical normalized output.

### 2. Canonical Section Ordering

State documents must be normalized into this section order when present:

1. `## Current State`
2. `## Open Questions`
3. `## Next Steps`

Do not invent missing sections in this spec.
Do not reorder content outside this canonical section order.

### 3. Canonical Bullet Normalization Rules

Within state documents, normalize bullet content as follows:

- remove exact duplicate bullets within the same section
- trim whitespace consistently
- collapse trivial formatting noise
- preserve meaningful bullet text
- preserve existing bullet order when that order is clearly intentional

If order is not clearly intentional, use a deterministic stable ordering strategy.

The normalization must not:
- invent new bullets
- silently merge semantically different bullets
- rewrite meaning for stylistic reasons only

### 4. Current State Rules

`## Current State` must represent present-tense, currently true facts.

The normalization path must support cleanup of obvious non-canonical items such as:
- completed-task checklist language
- stale implementation-history narration that does not describe current reality
- duplicated acceptance-criteria-style bullets repeated as present state

Keep this conservative:
- remove or normalize only when the text is clearly non-canonical noise
- do not rewrite legitimate current-state facts

### 5. Open Questions Rules

`## Open Questions` must contain unresolved questions only.

Normalization may:
- remove exact duplicates
- normalize formatting and spacing

Normalization must not:
- resolve questions
- move resolved questions here
- invent new questions

### 6. Next Steps Rules

`## Next Steps` must contain future work only.

Normalization may:
- remove exact duplicate bullets
- normalize formatting and spacing
- remove obvious already-completed items when they are clearly stale leftovers

Normalization must not:
- invent future work
- transform current-state facts into next steps

### 7. Determinism and Idempotency

Normalization must be deterministic and idempotent.

Given the same input:
- the same normalized output must always be produced

Given already normalized input:
- re-running normalization must produce byte-stable output or an equivalent stable output with no further changes

### 8. Integration Point

Integrate the normalization path into the most appropriate framework-owned context workflow.

Acceptable integration points include:
- a state-write path already used by context tooling
- a feature-state update path used during spec implementation
- a dedicated context utility invoked before state is persisted

Choose the smallest integration point that makes the behavior real and reusable.

### 9. Tests

Add focused coverage proving:

- duplicate bullets are removed deterministically within a section
- canonical section order is enforced when those sections are present
- already-normalized documents remain stable after re-normalization
- one-off ordering noise is normalized consistently
- `Current State` cleanup is conservative and does not rewrite legitimate current facts
- `Next Steps` cleanup removes obvious stale completed leftovers when present
- normalization does not invent content
- all relevant existing context tests still pass

## Non-Goals
- Do not normalize `docs/features/<feature>/<feature>.spec.md` in this spec.
- Do not normalize `docs/features/<feature>/<feature>.decisions.md` in this spec.
- Do not redesign context doctor or verify-context in this spec.
- Do not add user-configurable formatting rules in this spec.
- Do not broaden this into a repository-wide markdown formatter.

## Canonical Context
- Canonical feature spec: `docs/context-persistence/context-persistence.spec.md`
- Canonical feature state: `docs/context-persistence/context-persistence.md`
- Canonical decision ledger: `docs/context-persistence/context-persistence.decisions.md`

## Authority Rule
- Feature state documents must represent current reality in a canonical, deterministic structure.
- State normalization must preserve meaning while eliminating obvious ordering and formatting noise.
- Reusable normalization behavior must replace one-off manual cleanup for this class of issue.

## Completion Signals
- A reusable normalization path exists for feature state documents.
- Canonical section ordering is enforced for `Current State`, `Open Questions`, and `Next Steps` when present.
- Duplicate and obviously stale bullets are normalized conservatively and deterministically.
- Re-running normalization does not produce unstable diffs.
- Existing context behavior remains intact.
- All tests pass.

## Post-Execution Expectations
- Feature state documents are easier to keep clean and consistent.
- Context workflows produce less formatting and ordering drift.
- Later drift-detection work can rely on cleaner, more canonical state inputs.
