# Execution Spec: 015.002-context-doctor-diagnostic-rule-structure

## Feature
- context-persistence

## Purpose
- Introduce a cleaner internal rule structure for feature-scoped context diagnostics without breaking the current `context doctor` and `verify context` output contracts.
- Make `EXECUTION_SPEC_DRIFT` and future context-doctor checks easier to add consistently.
- Reduce ad hoc rule logic inside command and service code while preserving deterministic behavior.

## Scope
- Define an internal normalized rule-evaluation shape for context diagnostics.
- Keep external `context doctor` output in its current file-bucket structure.
- Keep external `verify context` output in its current flattened `issues` structure.
- Refactor only as much as needed to make `EXECUTION_SPEC_DRIFT` and future rules cleaner to implement.
- Keep this focused on internal rule structure, not on adding new user-facing policy beyond what the rule model requires.

## Constraints
- Preserve current external command behavior except where additive clarity is needed.
- Keep JSON contracts stable and deterministic.
- Keep diagnostics feature-scoped.
- Keep severity/proceed/repair semantics deterministic.
- Reuse existing doctor and verify-context infrastructure where practical.
- Do not redesign the entire context subsystem.
- Do not invent a second competing external diagnostics format.

## Requested Changes

### 1. Introduce an Internal Context Diagnostic Rule Model

Create or formalize an internal rule model for feature-scoped context diagnostics.

A rule should encapsulate:
- a stable issue code
- a deterministic evaluation method
- the conditions under which an issue is emitted
- the target canonical file bucket(s) the issue should attach to
- a stable message
- any required repair actions that should be surfaced at the top level

This may be implemented as a class, value object, interface, or similar internal abstraction, whichever fits the current architecture most naturally.

### 2. Define a Normalized Internal Rule Result

Each rule must be able to produce a normalized internal result shape sufficient for both doctor and verify-context.

The internal result should support, at minimum:
- `code`
- `message`
- one or more target file buckets (`spec`, `state`, `decisions`)
- one or more target file paths
- zero or more required actions

This internal result does not need to be exposed directly as a new external JSON contract.

### 3. Separate Rule Evaluation from External Rendering

Keep rule evaluation distinct from:
- `context doctor` JSON shaping
- `context doctor` plain-text rendering
- `verify context` issue flattening
- top-level proceed/repair summary logic

Rule evaluation should answer:
- whether an issue exists
- what the normalized internal result is
- which file bucket(s) it belongs to
- which required actions it contributes

Rendering and command-level summarization should consume those normalized results instead of embedding rule-specific conditionals.

### 4. Use `EXECUTION_SPEC_DRIFT` as the Reference Rule

Refactor the `EXECUTION_SPEC_DRIFT` implementation to use the new internal rule structure.

This rule should become the reference example for adding future feature-scoped doctor checks.

### 5. Preserve Current External Contracts

`context doctor --json` must continue to expose:
- top-level status/proceed/repair fields
- `files.spec`, `files.state`, `files.decisions`
- per-file `issues`
- top-level `required_actions`

`verify context --json` must continue to expose:
- top-level status/proceed/repair fields
- flat `issues`
- top-level `required_actions`

The new internal rule model must feed those existing contracts instead of replacing them.

### 6. Centralize Doctor-to-Verify Mapping

Verify-context must consume doctor-originated issues consistently.

At minimum, the internal structure must support deterministic mapping from doctor rule results to:
- per-file doctor issues
- flattened verify issues with `source = doctor`
- top-level `can_proceed`
- top-level `requires_repair`
- overall verify status

Prefer a central mapping strategy rather than rule-specific verify branching.

### 7. Preserve Deterministic Ordering

When multiple diagnostics are produced:
- ordering must be deterministic within file buckets
- ordering must be deterministic in verify-context flattened issues
- the same filesystem and document state must produce the same ordered results

If explicit sorting is needed, define it centrally.

### 8. Minimize Future Rule Cost

The design should make future rules cheap to add, such as:
- missing canonical context variants
- malformed context sections
- stale state-document structure
- other execution-spec misuse or drift variants

A future rule should ideally be implemented by adding one rule definition plus tests, not by editing multiple command-level conditionals.

### 9. Tests

Add focused coverage proving:

- rules produce normalized internal results consistently
- `EXECUTION_SPEC_DRIFT` works through the new rule model
- doctor output shape remains unchanged
- verify-context output shape remains unchanged
- doctor and verify ordering remain deterministic
- multiple rules can coexist cleanly
- no regressions occur in existing doctor and verify-context behavior
- all tests pass

## Non-Goals
- Do not redesign unrelated CLI commands.
- Do not add third-party extension loading for doctor rules.
- Do not change execution-spec naming or allocation rules.
- Do not add a new public diagnostics schema for doctor or verify-context.
- Do not invent a second competing diagnostics format.
