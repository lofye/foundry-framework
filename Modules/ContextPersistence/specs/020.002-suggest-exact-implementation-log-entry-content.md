# Execution Spec: 020.002-suggest-exact-implementation-log-entry-content

## Feature
- context-persistence

## Purpose
- Reduce implementation-log mistakes by having Foundry produce the exact canonical log entry content that should be appended for a completed active execution spec.
- Make implementation-log guidance explicit, deterministic, and machine-consumable.
- Complement 020.001 by helping humans and agents write the required log entry correctly the first time.

## Scope
- Add a deterministic way to generate or surface the exact implementation-log entry content for an active execution spec.
- Keep the existing implementation-log format canonical.
- Keep this focused on suggestion/generation of log-entry content, not on redesigning log enforcement or execution lifecycles.

## Constraints
- Keep the generated suggestion deterministic.
- Do not invent alternate log formats.
- Do not weaken the exact-match enforcement introduced by 020.001.
- Do not require LLM interpretation to derive the entry format.
- Reuse existing canonical spec identity and implementation-log rules where practical.
- Prefer a small, explicit generation path over a broad templating system.

## Inputs

Expect inputs such as:
- an active execution spec path like `docs/features/<feature>/specs/<id>-<slug>.md`
- canonical implementation-log format from `docs/features/README.md`
- exact spec identity derived from filename and feature path

If any critical input is missing:
- fail clearly and deterministically
- do not guess feature or spec identity
- do not emit non-canonical placeholder output

## Requested Changes

### 1. Add a Deterministic Log-Entry Suggestion Surface

Add a deterministic way to produce the exact implementation-log entry content for a given active execution spec.

Acceptable surfaces include:
- a dedicated CLI command such as:
  - `foundry spec:log-entry <feature> <id> --json`
  - or `foundry spec:log-entry <feature>/<id>-<slug> --json`
- or another equally explicit CLI-owned surface

The surface must output the canonical log entry content, not a paraphrase.

### 2. Emit Exact Canonical Entry Content

The suggestion must use the exact established log shape from `docs/features/README.md`, including:

- section heading timestamp format
- exact `- spec: <feature>/<id>-<slug>.md` line
- optional note shape only if such notes are already part of the canonical log rules

At minimum, the command must surface the exact spec line content the log requires.

If the timestamp is generated by the command, it must use the canonical timestamp format deterministically.

### 3. Preserve Exact-Match Compatibility with 020.001

The suggested/generated entry content must be fully compatible with the exact-match implementation-log enforcement added in 020.001.

That means:
- if a user or agent uses the suggested content correctly, `spec:validate` should accept the resulting log entry
- no formatting mismatch should be introduced between suggestion output and validation rules

### 4. Support Active Specs Only

This suggestion surface should operate on active execution specs only.

Do not suggest implementation-log entries for:
- draft specs under `docs/features/<feature>/specs/drafts/`

If the target is draft-only, fail clearly and explain that draft specs do not require implementation-log coverage.

### 5. Provide Machine-Readable Output

If `--json` is supported, the output should include deterministic fields such as:
- canonical spec ref
- canonical spec path
- exact spec log line
- full suggested log entry content

Choose the smallest JSON shape that is explicit and useful for agents.

### 6. Keep Existing Lifecycle Boundaries Intact

This spec does not require automatic log appending.

It only requires deterministic generation/suggestion of the exact canonical content so humans and agents do not have to reconstruct it from memory or repo docs.

### 7. Help and Documentation

Update:
- CLI help text
- any relevant registry/surface metadata
- `docs/features/README.md` if needed

Document:
- how to request the exact log entry content
- that the output is intended to match spec-validation enforcement exactly

### 8. Tests

Add focused coverage proving:

- the suggestion surface resolves an active spec correctly
- the suggested spec line exactly matches implementation-log enforcement
- draft specs fail clearly
- malformed/unknown ids fail clearly
- JSON output is deterministic
- repeated runs produce stable output, except for timestamp if intentionally generated at runtime
- all relevant CLI and validation tests still pass

## Non-Goals
- Do not redesign implementation-log format.
- Do not auto-backfill missing historical log entries in this spec.
- Do not auto-append log entries in this spec.
- Do not broaden this into a generic changelog generation system.
- Do not change active/draft lifecycle rules.

## Canonical Context
- Canonical feature spec: `docs/context-persistence/context-persistence.spec.md`
- Canonical feature state: `docs/context-persistence/context-persistence.md`
- Canonical decision ledger: `docs/context-persistence/context-persistence.decisions.md`

## Authority Rule
- Implementation-log entry content must be derivable deterministically from canonical spec identity and the established log format.
- Suggested/generated log-entry content must match validation expectations exactly.
- Draft specs remain exempt.

## Completion Signals
- Foundry can output the exact canonical implementation-log entry content for an active spec.
- Suggested content is accepted by implementation-log validation when used correctly.
- Draft and invalid targets fail clearly.
- Output is deterministic and machine-consumable.
- All tests pass.

## Post-Execution Expectations
- Codex and humans no longer have to reconstruct implementation-log entries from memory.
- Logging errors become less likely even before validation runs.
- The implementation-log workflow becomes more explicit, reliable, and automation-friendly.
